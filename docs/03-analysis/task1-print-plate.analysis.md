## 0. 요약 (Executive Summary)

현재 구현물은 **전체 시스템이 아니라 `spec.md` 공통 파이프라인 중 "정합(Registration) + 픽셀 차분" 원리만 검증하는 프로토타입**이다(README 스스로 그렇게 범위를 한정). 따라서 설계 문서(기획서 P0~P1 기능요구 + 검사자 UI 설계) 전체를 기준으로 하면 정합률은 낮고, 프로토타입이 스스로 선언한 범위를 기준으로 하면 대부분 충족한다. 두 기준을 모두 보고한다.

| 기준 | Match Rate | 해석 |
|---|---|---|
| **전체 설계 범위** (P0 9 + P1 6 기능요구) | **약 23%** | 대부분의 기능요구(OCR, 버전게이트, HITL, MES, UI)가 미구현 — 의도된 범위 축소 |
| **프로토타입 선언 범위** (정합+차분 원리검증) | **약 85%** | 원리 검증은 달성. 단, 치명 클래스 "미검출 0" 검증은 QA에서 반증됨 → 코드 수정으로 대응 |

가장 중요한 결론: **치명 클래스(성분표시오류) "미검출 0"이라는 최우선 성공기준은 아직 실질적으로 검증되지 않았다.** QA가 원인(Otsu 임계값 불안정)을 밝혔고, README 버그수정 이력이 이를 강건 임계값 + 오검출 가드레일로 대응했다고 기록하나, 이 수정 자체를 재검증할 후속 QA는 아직 없다.

## 3. QA 리포트와 현재 코드의 상태 차이 (중요)

QA 리포트(2026-07-07)가 지적한 3대 결함은 README 버그수정 이력상 **코드에서 대응 완료**로 기록되어 있다. 즉 QA 시점과 현재 코드 상태가 다르다.

| QA 발견 | 대응(README 기록) | 재검증 필요 |
|---|---|---|
| 발견1 Otsu 불안정 → 전면 오검출 | `robust_threshold`(median+k·MAD) + 오검출 면적 가드레일(20%) | **예** — 수정판을 독립 QA로 재확인 필요 |
| 발견2 라인단위 GT 근사 | 결함단위 bbox 측정(`_measure_substring_bboxes`) | 예 |
| 발견3 Precision 지표 부재 | `compute_precision_proxy` 추가 | 예 |

→ **권고: 이 수정본에 대한 재-QA를 다음 사이클에 반드시 배치.** README의 수정 주장(성분표시오류 단독 발생 시 박스 0/국소 검출)이 실제로 재현되는지 검증되어야 G1이 해소된다.

## 4. 권고 (다음 사이클 우선순위)

1. **[G1/재검증]** 수정된 robust threshold + 가드레일에 대한 재-QA: 성분표시오류 단독·무결함 케이스에서 오검출 면적비가 상한 이내인지 실측.
2. **[G2]** OCR 결합(Tesseract 설치 후): 점누락 등 미세결함 Recall을 Diff-only 대비 A/B 검증 — `spec.md` "정합+OCR+Diff" 설계 완성.
3. **[G4]** 버전 검증 게이트 최소 구현: 정합 inlier 수/버전 메타데이터로 불일치 감지(오검출 주원인 차단).
4. **[G5]** 실제 인쇄물/스캔 이미지 확보 후 파라미터 전면 재튜닝 및 합성-실측 갭 측정.
5. **[G6~G8]** 파일럿 단계에서 검사자 UI·HITL·MES 연계를 설계(UI 문서) 기준으로 구현 착수.

## 5. 정합률 산출 근거

- 전체 설계 범위: P0 9항목(충족0·부분4[F-01,F-03,F-06,F-08]·미충족5), P1 6항목(충족1[F-11]·부분1[F-10]·미충족4). 부분=0.5 가중 시 (2+1.5)/15 ≈ **23%**.
