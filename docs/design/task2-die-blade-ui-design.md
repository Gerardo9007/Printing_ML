# UI/UX 설계 — 과제② 목형 칼날검사 결과 확인 화면

> 근거 문서: `intro.md`, `problem.md`, `spec.md`, `prompt.md`, `docs/architecture/system-architecture.md`.
> 본 화면은 아키텍처 문서의 **"Verdict & Overlay Renderer"** 출력을 검사자가 확인하고, **"HITL Review Console"** 에서 재검을 확정하는 프런트엔드다. 이 문서와 상충되는 임의 가정은 만들지 않았으며, 필요한 가정은 "가정:"으로 표기했다.

## 0. 설계 전제 재확인 (설계 근거)

| 전제 | 근거 |
|---|---|
| 검사자는 **비개발자**다. 전문용어 없이 "도면과 무엇이 다른지"를 시각적으로 봐야 한다 | prompt.md 목표 5 |
| **끊김**(절단 불량)은 최우선 치명 클래스 — 미검출 0 목표. AI가 단독으로 "정상" 확정하지 않고 사람 재검을 거친다 | spec.md 2절, prompt.md 제약사항, architecture.md 0/2.1절 |
| **마모**는 이진(정상/불량)이 아니라 연속량 → 등급(정상/주의/교체) 또는 회귀값으로 표시 | spec.md 2절, 3.2/4절 |
| 판정 결과는 **불량종류 + 세그멘테이션/좌표 + mm 단위 위치오차**를 포함해야 함 | spec.md 1절·5절 |
| 오검출(과검출)은 재검으로 흡수 가능 → UI는 "일단 다 보여주고 사람이 걸러낸다"는 방향으로 설계 (과검출을 숨기지 않음) | prompt.md 목표 1 |
| 결과는 목형ID·제품·누적 사용횟수(타발수량)와 함께 추적 가능해야 함 (MES 연계) | spec.md 3.1, 7절 |
| 원본(CAD) 버전과 생산 버전 불일치는 "모델 결함"이 아니라 "데이터 관리 문제"로 별도 구분 표시 | spec.md 7절, architecture.md 2.1/3절 |
| 완전 무인 자동판정은 Non-goal → 화면은 항상 "AI 제안 → 사람 확정" 구조 | prompt.md Non-goals |

## 1. 전체 화면 흐름 (User Flow)

```

 │ (목형ID스캔)│     │ (지그 연동/수동) │     │ (diff+mm 시각화)     │     │ (승인/반려/보류)     │
 └────────────┘     └────────────────┘     └─────────┬───────────┘     └─────────┬───────────┘
                                                       │                          │
                                              치명(끊김) 검출 시                    │
                                              강제 재검 경로로 분기 ───────────────▶│
                                                       │                          ▼
                                                       │                 ┌────────────────────┐

## 2. 화면 ① — 대상 등록 (목형ID 확인 · 버전 게이트)

**목적**: 무엇을 검사하는지 특정하고, 기준 도면(CAD) 버전이 생산 버전과 일치하는지 먼저 확인한다. (spec.md 7절 "버전 관리" — 정합 이전 선행)

```
┌──────────────────────────────────────────────────────────────────┐
│  목형 칼날검사                                     [검사자: 홍길동] │
├──────────────────────────────────────────────────────────────────┤
│                                                                    │
│   목형ID 스캔/입력   [ MK-2026-04512        ]  [ 바코드 스캔 📷 ]   │
│                                                                    │
│   제품명            : 수분크림 50ml 외박스                          │
│   기준 도면(CAD) 버전: v3.2  (등록일 2026-05-10)                    │
│   현재 생산 지시 버전: v3.2                     ✅ 버전 일치         │
│   누적 사용(타발) 횟수: 128,400 회                                  │
│                                                                    │
│   ※ 버전이 다르면 검사를 시작할 수 없습니다.                          │
│      (→ "데이터 관리 이슈"로 분리 처리, 모델 오류 아님)               │
│                                                                    │
│                                       [ 다음: 촬영 진행 ▶ ]         │
└──────────────────────────────────────────────────────────────────┘
```

- 버전 불일치 시: 검사 진행 버튼 비활성화 + 빨간 배너 "도면 버전 불일치 — 생산관리팀 확인 필요" 표시. 이 상태는 AI 판정 실패와 명확히 분리해서 안내한다(architecture.md 2.1절 근거).
- 가정: 목형ID는 바코드/QR 태깅이 되어 있다고 가정(spec.md 메타데이터 항목 "목형ID" 기준). 실제 태깅 방식은 현장 확인 필요.

**목적**: 표준화된 지그 촬영 결과를 받거나(1차 권장), 예외적으로 파일 업로드를 허용한다. (spec.md 3.1 "촬영 표준화" — 조도/해상도 불일치 시 입력 거부)

```
┌──────────────────────────────────────────────────────────────────┐

│   ┌───────────────────────────────┐   촬영 지그 상태               │
│   │                               │   ● 조명(백라이트)   정상        │
│   │      [ 실시간 프리뷰 영상 ]     │   ● 거리/각도        정상        │
│   │                               │   ● 초점              정상        │
│   │                               │                                │
│   └───────────────────────────────┘   [ 📸 촬영 ]                  │
│                                                                    │
│   또는  [ 파일에서 업로드... ]  (지그 미연동 시 예외 경로)            │
│                                                                    │
│   ⚠ 초점 불량/과노출 감지 시 자동으로 재촬영을 요청합니다.            │
│                                                                    │
│                                       [ ◀ 이전 ]  [ 분석 시작 ▶ ]   │

- 지그 상태 3개 지표가 하나라도 "이상"이면 "분석 시작" 비활성화 → 촬영 표준화 미충족 데이터로 모델을 돌리지 않는다는 원칙(prompt.md 제약사항)을 화면 단에서 강제.
- 분석 시작 후에는 진행률 표시("정합 중 → 형상 비교 중 → 결과 생성 중")로 비개발자에게 "지금 뭘 하고 있는지"를 노출.

### 4.2 불일치 시각화 방식 ① — 원본 대비 diff 하이라이트

- **오버레이 방식**: 기준 도면(CAD) 외곽선은 **초록색 실선**, 실측 목형 칼날 외곽선은 **빨간색 실선**으로 동시에 그려 겹친다. 두 선이 완전히 겹치면 "일치"이므로 화면상 초록만 보이고, 어긋난 구간만 빨간선이 삐져나와 시각적으로 즉시 눈에 띈다.
- **3-way 토글**: `[원본]`(도면만) / `[겹침]`(오버레이, 기본값) / `[실물]`(촬영 이미지만) 버튼으로 전환 — 검사자가 "어긋난 부분이 도면 기준으로 어디인지" vs "실물이 실제로 어떻게 생겼는지"를 각각 확인 가능.
- **투명도 슬라이더**: 실측선의 불투명도를 조절해 두 선이 겹칠 때도 구분이 쉽도록 함.
- **결함 유형별 색상/아이콘 고정 규칙** (범례를 화면 하단에 항상 노출):

| 유형 | 색상 | 아이콘 | 비고 |
|---|---|---|---|
| 끊김(절단 불량) | 🔴 빨강 | ⚠ | 치명 — 항상 최상단 정렬 |
| 휨 | 🟠 주황 | 〰 | |
| 위치오차 | 🟡 노랑 | ⇄ | |
| 마모 | 🟢/🟡/🔴 (등급별) | ● | 등급 표시(4.3절) |

- 각 결함 위치에는 番号 배지(①②③④)를 달아, 오른쪽 리스트의 항목과 1:1 매칭시킨다. 배지를 클릭/탭하면 해당 부위로 자동 확대(zoom)한다.
- **줌 인/아웃**과 **핀치 확대**(터치 지원)로 미세 결함(마모 등)도 실측 가능하게 한다.

### 4.3 불일치 시각화 방식 ② — mm 단위 오차 표시

비개발자가 "3픽셀 어긋남" 같은 표현을 이해하기 어렵다는 점을 감안해, **항상 mm(실측 단위)로 환산해 표시**한다 (spec.md 5절 "위치오차(mm)" 요구사항).

```
        기준선(도면, 초록) ─────────────────
                                 ╲
                                  ╲   ←── 실측선(빨강)이 바깥으로 밀림
                                   ╲
                            ┌───────────────┐
                            │  +0.42 mm     │  ← 캘리아웃(말풍선) 라벨
                            │  (허용 ±0.30) │     허용공차 대비 초과분 병기
                            └───────────────┘
```

- 결함 지점마다 **치수선(dimension line) + 캘리아웃**으로 "도면 대비 몇 mm 벗어났는지"를 화살표와 숫자로 직접 표기한다 (CAD 도구에서 흔히 쓰는 치수 표기 방식이라 현장 도면 문화와 익숙함).
- 색상으로 심각도 구간을 구분(허용공차는 spec.md에 수치가 명시되지 않아 **가정**으로 표기):
  - 🟢 허용범위 이내 (예: 표시는 하되 "정상" 처리)
  - 🟡 주의 (허용공차 초과, 경미)
  - 🔴 초과 (허용공차 대비 큰 폭 이탈 또는 치명 클래스)
  - **가정**: 정확한 mm 공차 기준값은 spec.md/problem.md에 수치가 없어 미정 — 실제 값은 품질기준 문서 확인 후 설정 필요 (확인 필요 항목으로 하단 6장에 재기재).
- **마모(연속량) 표시는 mm 오차 캘리아웃 대신 게이지(등급) 위젯 사용**:

```
   마모도  ─────────────────────────
   정상 ▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░  주의  ────  교체
                    ▲
              현재 상태: 주의
        (누적 타발 128,400회 / 권장 교체 150,000회)
```

  이진 합격/불합격이 아니라 슬라이더형 게이지 + 등급 라벨(정상/주의/교체)로 표시해, 연속량이라는 spec.md 특성을 UI에 그대로 반영한다.

### 4.4 판정 요약 카드 규칙

- 검출 항목 중 **치명 클래스(끊김)가 하나라도 있으면 종합 판정은 무조건 "재검 필요"로 고정**되며, 이 상태에서는 AI가 "정상" 버튼을 노출하지 않는다(architecture.md "치명 클래스 검출 시 강제 재검 라우팅" 그대로 UI화).
- 치명 결함이 없고 경미한 항목만 있을 때도, 종합 판정은 "AI 제안: 정상" 처럼 **제안 문구**로만 표시하고 최종 확정 버튼은 항상 사람이 누르게 한다(Non-goal: 완전 무인 자동판정 아님).

## 5. 화면 ④ — 재검 처리(확정)

**목적**: 검사자가 AI 제안을 확인하고 최종 승인/반려/보류를 결정. 치명 클래스는 반드시 사유를 남기도록 강제.

```
┌──────────────────────────────────────────────────────────────────┐
│  재검 처리                                목형ID: MK-2026-04512    │
├──────────────────────────────────────────────────────────────────┤
│  검출: 🔴 끊김 (좌측 상단 모서리)                                    │
│  [ 결함 이미지 확대 보기 ]   [ mm 오차 상세 보기 ]                    │
│                                                                    │
│  검사자 판단                                                        │
│   ( ) 승인 — AI 판정에 동의, 재작업/폐기 처리                        │
│   ( ) 반려 — AI 오검출로 판단, 정상 처리                             │
│   ( ) 보류 — 추가 확인 필요 (2차 검사자 배정)                        │
│                                                                    │
│  판단 사유 (필수, 치명 클래스는 미입력 시 제출 불가)                  │
│   [ 자유 입력란                                              ]      │
│                                                                    │
│  ※ 치명 클래스(끊김)는 AI 단독으로 "정상" 확정할 수 없으며,          │
│     반드시 검사자 확정 사유가 기록됩니다.                            │
│                                                                    │
│                                    [ 취소 ]   [ 확정 제출 ✔ ]       │
└──────────────────────────────────────────────────────────────────┘
```

- 상태 머신: `AI 제안` → (검사자) → `승인/반려/보류` → `Traceability Store 기록`. 이 화면이 architecture.md의 "HITL Review Console" 승인/반려/보류 상태 머신을 그대로 구현한다.
- "반려(오검출)"를 선택해도 기록은 남긴다 — 오검출 통계를 추적해 향후 모델 개선(리얼-합성 갭 모니터링, spec.md 8절)에 활용하기 위함.
- 보류 선택 시 2차 검사자 배정 큐로 이동(이중검수 원칙, spec.md 5절 "2인 교차검수"와 사상 일치 — **가정**: 재검 단계에도 이중검수 개념을 적용, spec.md는 "라벨링" 단계 이중검수만 명시하므로 검사 확정 단계 적용은 확장 가정임을 표기).

## 7. 비개발자 이해를 돕기 위한 설계 원칙 (요약)

1. **전문용어 대신 색+화살표+숫자(mm)**: "IoU 0.62" 같은 지표는 검사자 화면에 노출하지 않는다. 대신 "도면보다 0.42mm 밖으로 나감"처럼 실측 단위로 번역해 보여준다.
2. **항상 "AI 제안" vs "사람 확정"을 분리 표기**: 버튼/문구에서 AI가 낸 결과는 "제안", 검사자가 누른 결과만 "확정"이라는 단어를 써서 책임 소재를 명확히 한다.
3. **치명 클래스는 색상·정렬 우선순위·강제 경로로 3중 강조**: 목록 최상단 고정 + 빨간색 + AI가 "정상" 버튼을 아예 숨기는 방식으로 미검출 0 목표를 화면 레벨에서 강제.
4. **마모처럼 연속적인 결함은 합격/불합격 대신 게이지**로 표현해 이진 판정의 오해를 방지한다.
5. **도면 버전 불일치는 별도 배너로 분리**하여 "모델이 틀렸다"는 오해 없이 원인(데이터 관리)을 바로 알 수 있게 한다.

## 8. 확인 필요 / 후속 검토 (임의 확정하지 않음)

- mm 오차 허용공차(정상/주의/초과 구간 경계값)의 실제 수치 — spec.md·problem.md에 명시 없음. 품질기준 문서 확보 후 4.3절 색상 구간에 반영 필요.
- 재검 단계 이중검수(보류→2차 검사자) 적용 여부 — spec.md는 "라벨링" 단계 이중검수만 명시. 실제 현장 재검 프로세스에도 이중검수를 적용할지는 확인 필요.
- 목형ID 태깅 방식(바코드/QR/수기입력) — spec.md는 "메타데이터: 목형ID" 항목만 명시, 실제 식별 수단은 미기재.
- 엣지 vs 서버 추론 확정 전이므로(architecture.md 4절), 촬영→분석 소요 시간(진행률 UX)의 실제 체감 지연은 인프라 결정 이후 재검토 필요.
