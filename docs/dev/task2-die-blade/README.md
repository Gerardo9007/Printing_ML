# 과제② 목형 칼날검사 — 파이프라인 원리 검증 프로토타입

이 폴더는 `spec.md` 1절의 공통 파이프라인

```
[촬영/입력] -> [전처리·정규화] -> [정합(Registration)] -> [차분/이상탐지] -> [판정·시각화] -> [결과 기록]
```

을 **목형 칼날검사(과제②)**에 좁혀, 실제 목형 사진/CAD 파일 없이도 원리가 동작함을 보이는 실행 가능한 데모다.
`problem.md`가 지적하는 "실불량 실물 확보가 근본적으로 어렵다"는 문제를 우회하기 위해, `spec.md` 3.2절
"합성 결함 삽입(국소 끊김·휨)" 기법을 코드로 직접 구현했다 — 기준 도면(CAD)에 해당하는 합성 사각 칼날
윤곽을 만들고, 그 위에 결함(휨/끊김)을 주입해 "실물처럼 보이는" 버전을 만든 뒤, 정합 + diff로 결함을
검출한다.

## 가정 (Assumptions) — 반드시 확인

`spec.md`/`problem.md`/`intro.md`에 수치가 명시되지 않아, 데모 구현을 위해 아래 값을 **임의로 가정**했다.
실제 배치 시에는 지그·CAD 스펙 기준으로 재조정되어야 한다.

- **가정**: 1px = 0.2mm (촬영 스케일 캘리브레이션). 실제 값은 촬영 지그·렌즈 스펙에서 결정된다.
- **가정**: 결함 판정 허용오차(정합 잔차 포함) = 1.0mm(=5px). 목형 공차 스펙에 따라 조정 필요.
- **가정**: 최소 결함 인정 길이 = 2.0mm 미만의 편차는 노이즈/정합 잔차로 간주해 무시(과검출 억제).
- **가정**: 목형은 백라이트 실루엣 촬영으로 이미 이진(선/배경) 이미지 형태로 얻어졌다고 가정한다
  (`spec.md` 3.1 "백라이트(실루엣) 또는 텔레센트릭 렌즈로 에지 캡처"). 즉 이 데모는 촬영 이후
  단계(정합~판정)만 검증하며, 실제 카메라 촬영·조명 보정 단계는 다루지 않는다.
- **범위 제외**: `spec.md`가 별도 취급을 요구하는 "마모"(연속량 → 회귀/등급화 필요)와 "위치오차"는
  이번 데모의 핵심 검증 대상(휨·끊김, 최우선 치명 클래스)에서 제외했다. 마모는 이진 판정이 아니라는
  제약(`prompt.md` 한계사항)을 고려해 별도 프로토타입에서 다루는 것이 맞다고 판단했다.

## 파이프라인 구현 방식

1. **기준 도면 생성** — 모서리가 둥근 사각형 다이라인을 조밀한 점열(폐곡선)로 합성 생성 (`generate_reference_contour`).
2. **결함 주입** (`spec.md` 3.2 합성 결함 삽입)
   - **휨**: 곡선 일부 구간을 Hann 창(부드러운 종형 곡선)으로 바깥쪽 법선 방향으로 밀어냄 (`inject_bend`).
   - **끊김**: 곡선 일부 구간의 점을 완전히 제거해 실제로 선이 끊어지도록 렌더링 (`inject_break`).
3. **촬영 정렬오차 시뮬레이션** — 목형이 지그에 완벽히 놓이지 않는 상황을 흉내내기 위해 결함 주입 후
   전체 이미지에 미세한 회전+이동을 추가로 적용 (`apply_camera_misalignment`).
4. **정합(Registration)** — 얇은 선 이미지는 특징(그래디언트)이 희박해 정합이 불안정하므로, 두 이진
   마스크를 두껍게 팽창(dilate)한 버전으로 OpenCV `findTransformECC`(MOTION_EUCLIDEAN)를 이용해
   회전+이동 변환을 추정한 뒤, 그 변환을 원본 얇은 실물 마스크에 적용해 기준 도면 좌표계로 정렬한다.
5. **차분(diff) 기반 결함 검출** — `cv2.distanceTransform`으로 두 방향의 최근접 거리장을 계산한다.
   - 기준 도면 픽셀 기준으로 "실물에서 가까운 픽셀이 없는 곳"(거리 > 허용치) → **끊김** 후보
   - 실물 픽셀 기준으로 "기준 도면에서 가까운 픽셀이 없는 곳"(거리 > 허용치) → **휨** 후보
   - 연결요소 분석(`connectedComponentsWithStats`)으로 픽셀을 결함 단위 영역으로 묶고, bbox·최대 편차(mm)·
     대략적 결함 길이(mm)를 산출한다. 최소 길이 미만은 노이즈로 간주해 제외한다.
6. **시각화·기록** — 기준 도면(회색)·정합된 실물(녹색)·결함 위치(빨강=끊김, 주황=휨) bbox와 mm 수치를
   PNG로 오버레이하고, 동일 정보를 JSON으로도 기록한다 (한글 라벨은 OpenCV Hershey 폰트가 한글을
   지원하지 않아 PIL + 맑은 고딕(`malgun.ttf`)으로 별도 렌더링).

## 알아둘 점 (관찰된 특성, 버그 아님)

- **휨 결함은 부수적으로 "끊김"도 함께 검출되는 경향이 있다.** 칼날이 바깥으로 휘면 원래 위치(기준 도면
  선)에는 더 이상 실물 선이 없으므로, 그 자리가 "기준 도면에는 있으나 실물은 없는 곳"으로도 잡혀
  끊김 후보로 동시에 검출된다. 이는 오탐이 아니라 "결함이 있는 위치"를 두 지표(끊김/휨)가 서로 다른
  각도에서 함께 가리키는 것으로, 미검출 0 목표(`prompt.md` 목표1)에는 오히려 유리하다.
  **[해결됨]** 이 현상을 유형 통계·재검 우선순위 혼선(Gap 목록 G5)으로 이어지지 않도록, 아래
  "분류 정교화" 절의 후처리(`resolve_bend_break_overlap()`)로 같은 위치에서 겹치는 휨/끊김을
  `"복합(휨+끊김 의심)"`으로 명시 태깅하도록 정교화했다.
- **한글 텍스트 렌더링**: `cv2.putText`는 Hershey 폰트만 지원해 한글이 물음표로 깨진다. 이 데모는
  PIL + 시스템 폰트(`C:\Windows\Fonts\malgun.ttf`)로 우회했다. 이 폰트가 없는 환경(Windows가 아닌
  환경)에서는 `_load_korean_font()`가 PIL 기본 폰트로 폴백하며, 이 경우 한글 라벨이 표시되지 않을 수
  있다 — "확인 필요: 비-Windows 실행 환경에서의 한글 폰트 경로".
- **한글 경로 이슈**: 이 프로젝트 경로 자체에 한글이 포함되어 있어 `cv2.imwrite()`가 무음 실패한다.
  `cv2.imencode` + `ndarray.tofile()` 조합으로 우회했다 (OpenCV의 알려진 Windows 유니코드 경로 제약).

## 확인 필요 (Verification Needed) — 실제 데이터 확보 후 재검증할 항목

- 이 데모는 완전한 합성 데이터로 파이프라인의 **원리**(정합 → diff → 판정)만 검증한 것이다.

  (`spec.md` 8절 "미해결·후속 검토 사항"과 연결).

## 버그 수정 이력

### [치명, 수정 완료] 동일 종류 결함이 2개 이상 있으면 렌더링 단계에서 일부가 소실되어 미검출되는 버그 (QA 리포트 §2-A)

QA 검증(`docs/qa/task2-die-blade-qa-report.md` §2-A) 과정에서, 동일 목형에 **끊김을 3개 동시에 주입**하면
검출이 1건만 되는(66% 미검출) 치명 버그가 발견되었다. spec.md 6절 핵심 목표("끊김·휨 미검출 0")에
정면으로 반하는 문제였다.

**근본 원인 (2가지)**

1. **폐곡선 랩어라운드 병합 로직의 오적용.** `inject_break()`에는 "폐곡선을 한 번 자르면 열린 호가
   되므로, 처음/끝 run이 로컬 인덱스 0과 n-1에서 만나면 하나로 이어붙인다"는 로직이 있다. 이 로직은
   원본이 **닫힌 폐곡선일 때의 최초 컷**에서만 유효하다. 그런데 `run_scenario()`가 두 번째·세 번째
   끊김을 **이미 열린 호(첫 컷의 결과물)**에 적용할 때도, 그 열린 호 자체의 로컬 인덱스가 마침 0과
   len-1에서 시작/끝나기 때문에 조건이 우연히 참이 되어 이미 분리되어야 할 두 조각을 다시 하나로
   이어붙였다(bridging). `render_polylines()`는 그 배열을 순서대로 직선 연결해 그리므로, 실제로는
   끊어져 있어야 할 자리가 시각적으로 다시 메워져 사라졌다.
2. **`start_frac`의 기준점이 매번 달라지는 문제.** `inject_break()`가 `start = int(start_frac * n)`을
   계산할 때 `n`이 "그 시점에 남아있는(잘려서 재인덱싱된) 조각의 길이"였다. 즉 두 번째·세 번째 결함의
   `start_frac`이 호출자가 의도한 "원본 도면 전체 둘레 기준 위치"가 아니라 "직전 컷 이후 남은 조각의
   로컬 위치"로 잘못 해석되어, 실제 주입 위치가 의도와 달라졌다.

**수정 방법 (근본 원인 대응)**

- 끊김 주입 도중 다루는 각 조각을 `Arc(points, orig_idx, is_closed)` 구조로 추적하도록 변경했다.
  - `orig_idx`: 각 점이 "아직 어떤 컷도 적용되지 않은 원본 폐곡선"에서 가졌던 인덱스. 조각이 잘려나가도
    이 값은 유지되므로, 위치 해석이 항상 원본 전체 기준으로 일관된다.
  - `is_closed`: 이 조각이 "아직 컷이 한 번도 적용되지 않은 원본 폐곡선 그 자체"인지 여부. 랩어라운드
    병합은 이 플래그가 `True`일 때(=그 곡선에 대한 최초 컷)만 수행하고, 한 번이라도 컷이 적용된 이후에는
    항상 `False`로 유지해 다시는 병합되지 않도록 했다.
- `inject_break(arc, start_frac, span_frac, n_total)`로 시그니처를 바꿔, `start_frac`을 항상 `n_total`
  (=원본 전체 폐곡선의 점 개수, `run_scenario()`에서 `len(reference_points)`로 고정)을 기준으로
  환산하도록 통일했다. 조각이 몇 번 잘려나갔든 같은 `start_frac` 값은 항상 같은 물리적 위치를 가리킨다.
- `run_scenario()`도 `Arc` 리스트를 다루도록 갱신하고, 렌더링 직전에 `[a.points for a in polylines]`로
  풀어서 기존 `render_polylines()`에 전달하도록 했다(렌더링 로직 자체은 변경하지 않았다).

### [신뢰성 개선] ECC 대각도 오수렴 감지 가드레일 (QA §2-B)

**문제**

`register_actual_to_reference()`는 `cv2.findTransformECC`(MOTION_EUCLIDEAN)로 실물 마스크를 기준
도면에 맞춰 회전+이동 정렬한다. 이 데모의 형상(모서리가 둥근 사각형)은 180° 근사 대칭을 가지므로,
촬영 정렬오차(회전)가 대략 40° 이상으로 커지면 ECC가 실제 회전이 아니라 "180° 근처 대칭에 의한
착각 정렬"이라는 전혀 다른 지역해에 수렴할 수 있다. 문제는 `findTransformECC`가 이 경우에도
예외 없이 `converged=True`를 반환한다는 점이다 — 즉 **정합이 완전히 틀렸는데도 코드상으로는
"정합 성공"으로 보고**되고, 그 잘못 정렬된 실물을 기준으로 diff를 계산하니 결함이 아닌 곳들이
무더기로 결함처럼 검출된다(QA 재현: 끊김 1건 주입 시 회전 45°에서 17건 오검출).

**수정 방법**

`registration_converged`(ECC 자체 보고값)와는 독립적으로, 정합 이후 **잔차 지표**를 별도로 계산해
정합이 실제로 두 형상을 겹쳐놓았는지를 검증하는 가드레일을 추가했다.

- 신규 함수 `compute_registration_residual(reference_mask, aligned_actual_mask)`: 정합된 두
  마스크 사이의 **대칭 평균 최근접거리**(symmetric mean chamfer distance)를 계산한다. 기존
  `detect_defects()`가 이미 사용하던 `cv2.distanceTransform`을 그대로 재사용하되, 개별 결함
  영역이 아니라 **전체 라인 픽셀**에 대한 평균을 낸다는 점이 다르다(정합 자체의 전반적 품질을
  보기 위함 — 국소적 결함과 전역적 오정렬을 구분).
  - 기준 도면의 각 선 픽셀 → 정합된 실물에서 가장 가까운 선 픽셀까지 거리의 평균
  - 정합된 실물의 각 선 픽셀 → 기준 도면에서 가장 가까운 선 픽셀까지 거리의 평균
  - 두 값의 평균을 최종 잔차로 사용(한쪽 마스크만 유난히 두껍거나 조각나 있는 경우의 편향을 줄이기 위함)
- 이 잔차(px)를 mm로 환산해 임계치 `REGISTRATION_RESIDUAL_TOL_PX = 10px`(`= 2.0mm`,
  `REGISTRATION_RESIDUAL_TOL_MM`)와 비교하고, `registration_reliable = registration_converged
  and residual <= threshold`를 새 필드로 추가했다. **`registration_converged` 값 자체는 그대로
  유지**하며(ECC가 실제로 보고한 값을 덮어쓰지 않음), 신뢰도 판단은 별도 필드로 노출한다.
- `registration_reliable=False`인 경우에도 **검출된 결함 리스트(`detected_defects`)는 삭제·억제
  하지 않는다** — QA 지침대로 "정합 실패/재검 필요"를 알리는 가드레일이지 결과를 숨기는 목적이
  아니기 때문이다. 대신 JSON 리포트, 콘솔 로그, PNG 시각화 세 군데 모두에 신뢰 불가 상태를
  명시적으로 노출해 사람이 재검할 수 있게 한다.
  - JSON: `registration_residual_px`, `registration_residual_mm`, `registration_residual_tol_mm`,
    `registration_reliable` 필드 추가 (`assumptions`에도 임계치 기록)
  - 콘솔: `정합 신뢰도(잔차 기반, QA §2-B 가드레일): OK` 또는 `*** 신뢰 불가(재검 필요) ***` 로그 출력
  - PNG: 신뢰 불가 시 이미지 상단에 빨간 경고 배너(`[경고] 정합 신뢰 불가 ... 재검 필요`)를 추가로 그림

**임계치(threshold) 선정 근거 — 실측**

QA §2-B 재현 시나리오(끊김 1개 주입 + 회전 오차만 1.4°/35°/45°/90°로 변경, 나머지 조건 동일)를
그대로 재현해 잔차를 실측했다(재현 스크립트는 저장소에 포함하지 않은 별도 검증용 스크립트이며,
`run_scenario()`를 그대로 호출해 얻은 실측값이다):

| 실제 회전 오차 | ECC 추정 회전 | registration_converged | 잔차(px) | 잔차(mm) | registration_reliable | 검출 개수 |
|---|---|---|---|---|---|---|
| 1.4° | -1.40° (정확) | True | 1.26 | 0.25 | **True** | 1 (정상) |
| 35° | -35.00° (정확) | True | 1.27 | 0.25 | **True** | 1 (정상) |
| **45°** | **11.03°** (완전히 틀림) | True | **49.39** | **9.88** | **False** | **17** (거짓 양성 폭증) |
| **90°** | **0.00°** (완전히 틀림) | True | **69.29** | **13.86** | **False** | **9** (거짓 양성 폭증) |

정상 정합 구간(1.4°~35°)의 잔차는 항상 ~1.3px 수준으로 매우 작고, 대칭 오수렴 구간(45°~90°)의
잔차는 49px 이상으로 약 **40배 이상 차이**가 나는 명확한 간극이 있다. 이 간극의 중간 어디에
임계치를 두어도 두 구간을 정확히 구분할 수 있으므로, 정상 구간에 충분한 여유를 두면서 오수렴
구간보다는 확실히 낮은 `REGISTRATION_RESIDUAL_TOL_PX = 10px`(`= 2.0mm`, 결함 판정 허용오차
`DEFECT_TOL_MM`의 2배 — "결함 허용오차보다는 크게 잡아 정상적인 정합 잔차/노이즈까지 오탐하지
않으면서도, 오수렴 구간의 최솟값(49px)보다는 5배 가까이 낮게" 선택)를 채택했다.

(QA 원 리포트의 90° 사례는 오검출 4건으로 기록되어 있으나, 이번 재현에서는 9건으로 다소 차이가
있었다 — 결함 위치·정합 초기조건 등 세부 실행 조건 차이로 보이며, "대각도에서 정합이 완전히
틀어지고 오검출이 정상 대비 폭증한다"는 핵심 현상 자체는 QA 리포트와 동일하게 재현되었다.)

**회귀 검증 (기존 case0~6)**

### [근거 보강] 최소 결함 길이 필터(`MIN_DEFECT_ARC_MM = 2.0mm`) 임계값 스윕 (QA §2-C)

**문제**

`detect_defects()`는 검출된 결함 후보의 bbox 긴 변이 `MIN_DEFECT_ARC_MM(2.0mm)` 미만이면 과검출
억제를 위해 무조건 노이즈로 간주해 버린다. 이 값은 코드 주석과 README 모두에서 "가정"으로만
명시되어 있었고, 왜 하필 2.0mm인지에 대한 실측 근거가 없었다(QA §2-C 지적).

**대응 방법**

값 자체를 바꾸는 것이 아니라(여전히 2.0mm 유지), 그 값 근처에서 실제로 검출 성능이 어떻게
바뀌는지를 코드로 스윕 테스트해 실측 근거를 남겼다.

- 신규 함수 `sweep_min_defect_arc_threshold(threshold_candidates_mm)`: 임계값 후보들에 대해
  두 가지를 실측한다.
  1. **과검출(false positive)**: 결함이 전혀 없는 정상 케이스(촬영 정렬오차만 존재, case0과
     동일 조건)에서 각 임계값에서 몇 건이 잘못 검출되는지.
  2. **미검출(false negative)**: 목표 길이 0.1mm~4.0mm의 인위적 끊김을 주입했을 때, 각
     임계값에서 실제로 검출되는지.
- 신규 헬퍼 `_inject_calibrated_gap()`: 기존 `inject_break()`는 데모 취지상 "최소 4점 제거"
  하한을 두고 있어(너무 짧은 끊김은 렌더링 두께에 묻혀 무의미하다는 전제), 이 하한 때문에
  0.5mm~1.5mm대 목표 길이가 전부 동일한(~1.8mm) 실제 길이로 뭉개져 스윕이 무의미해지는 문제가
  있었다. 스윕 전용으로 하한 없이 목표 길이에 정밀하게 맞춰 점을 제거하는 헬퍼를 별도로
  추가해 이 문제를 피했다.
- `main()` 실행 시 스윕을 자동으로 함께 실행하고, 콘솔 로그와 `output/min_defect_arc_sweep.json`에
  결과를 남기도록 했다.

**실측 결과 (실행 로그, `python die_blade_qc_demo.py`)**

| 임계값(mm) | 정상 케이스 과검출 | 실측 끊김 길이별 검출 여부 (0.1 / 0.3 / 0.5 / 1.0 / 1.5 / 2.0 / 2.5 / 3.0 / 4.0 mm) |
|---|---|---|
| 0.5 | 0건 | X, X, X, X, O, O, O, O, O |
| 1.0 | 0건 | X, X, X, X, O, O, O, O, O |
| 2.0 (현재값) | 0건 | X, X, X, X, O, O, O, O, O |
| 3.0 | 0건 | X, X, X, X, O, O, O, O, O |
| 5.0 | 0건 | X, X, X, X, X, X, X, X, O |
| 8.0 | 0건 | X, X, X, X, X, X, X, X, X |
| 10.0 | 0건 | X, X, X, X, X, X, X, X, X |

(O=검출, X=미검출. 목표값과 실측 길이는 이산화 때문에 정확히 일치하지 않는다 — 예: 목표 1.5mm는
실측 1.89mm, 목표 2.0mm는 실측 2.51mm로 주입됨. 상세 수치는 `output/min_defect_arc_sweep.json` 참고.)

**해석 — 솔직한 결론**

이번 스윕으로 확인된 것은 "2.0mm가 최적값"이라는 증거가 **아니라**, 다음과 같은 더 정확한 그림이다.

1. **0.5mm~3.0mm 범위 내에서는 `MIN_DEFECT_ARC_MM` 값이 실제 검출 결과에 전혀 영향을 주지
   않는다.** 실측 끊김 길이가 약 1.26mm 이하일 때는 이 범위의 어떤 임계값을 넣어도 미검출이고,
   약 1.89mm 이상일 때는 어떤 임계값을 넣어도 검출된다. 즉 진짜 결정 요인은 `MIN_DEFECT_ARC_MM`이
   아니라 **결함 판정 허용오차(`DEFECT_TOL_MM = 1.0mm`)와 라인 두께·팽창(dilate) 커널 기하학**이
   만드는 훨씬 더 급격한 절벽(1.26mm→1.89mm 사이)이다. `MIN_DEFECT_ARC_MM`은 이 절벽보다 훨씬
   작은 값이라면 사실상 아무 것도 걸러내지 않는다.
2. **`MIN_DEFECT_ARC_MM`이 실제로 결함을 놓치기 시작하는 지점은 대략 4~5mm 부근부터다**
   (임계값 5.0mm에서 실측 3.14mm 결함까지 미검출 처리됨). 현재 값 2.0mm는 이 위험 구간에서
   충분히 멀리 떨어져 있어, 적어도 이번에 테스트한 크기(1.5mm~4.0mm)의 결함을 이 필터 때문에
   놓칠 위험은 없다는 것을 확인했다.
3. **정상 케이스 과검출은 테스트한 모든 임계값(0.5mm~10.0mm)에서 0건이었다.** 이는 이 필터가
   "과검출 억제"라는 원래 설계 의도를 정말로 지금 이 합성 데이터에서 입증하는 것이 아니라,
   현재 완전 합성 데이터의 정합 잔차가 워낙 작아(§2-B 표 참고, 0.03mm~0.48mm)애초에 노이즈성
   과검출이 하나도 발생하지 않기 때문이다. 즉 이 스윕은 "과검출 억제" 목적에 대한 근거는 아직
   실측하지 못했다 — 실제 조명·센서 노이즈가 있는 데이터가 있어야 이 필터가 진짜로 방어하는
   시나리오(자잘한 노이즈 블롭)를 재현하고 임계값을 그에 맞게 검증할 수 있다.

**결론 및 남은 과제**

- `MIN_DEFECT_ARC_MM = 2.0mm`는 "1.5mm~4.0mm 크기의 결함을 놓치지 않으면서, 필터가 실제로
  위험해지는 4~5mm 구간과는 충분한 안전거리를 둔 값"이라는 근거를 이번에 확보했다. 다만 이는
  여전히 완전 합성 데이터(노이즈 없는 정합) 위에서의 결론이라는 한계가 있다.
- 진짜 실무적으로 재조정이 필요한 파라미터는 `MIN_DEFECT_ARC_MM` 자체보다 **`DEFECT_TOL_MM`
  (결함 판정 허용오차)와 라인 두께/팽창 커널 기하학**일 가능성이 이번 스윕으로 새롭게 드러났다 —
  이 두 파라미터가 실제 최소 검출 가능 결함 크기(1.26mm~1.89mm 절벽)를 결정하기 때문이다. 이는
  QA §3 "최소 결함 크기 기준 확정" 항목에 추가할 후속 확인 사항으로 남긴다.
- 실제 조명/센서 노이즈가 섞인 데이터를 확보한 뒤 이 스윕을 다시 돌려 "과검출 억제" 측면의 근거를
  보강해야 한다(현재는 이 측면을 합성 데이터의 한계로 인해 실측하지 못했다).

### [분류 정교화] 휨/끊김 동시검출 시 유형 정리 후처리 (Gap 목록 G5)

**문제**

앞서 "알아둘 점" 절에서 관찰로만 기록했던 현상 — 휨 결함이 발생하면 원래 위치의 선이 실물에서
사라지므로, 같은 위치가 "끊김" 후보로도 부수적으로 동시 검출되는 경향 — 이 Gap 분석에서
G5("휨/끊김 최종 분류 규칙 미정교화")로 정식 지적되었다. 미검출 0 목표에는 유리하지만, 실제
운영에서 결함 유형 통계(끊김 몇 건, 휨 몇 건)를 낼 때나 재검 우선순위를 정할 때 같은 결함이
두 유형으로 중복 집계되는 혼선이 있었다.

**대응 방법**

`detect_defects()`가 반환한 원본 검출 목록은 그대로 두고(결과 자체는 삭제·억제하지 않음 — 여전히
`spec.md` 6절 "미검출 0" 목표가 최우선이기 때문), 그 뒤에 신규 후처리 함수
`resolve_bend_break_overlap(defects)`를 추가해 유형만 정리한다.

- 휨 결함과 끊김 결함의 bbox가 실제로 겹치는 경우(=물리적으로 같은 위치)에만 적용한다. 겹치지
  않는 별개의 휨과 끊김(예: case3처럼 서로 다른 위치에서 진짜로 각각 발생한 결함)은 손대지
  않는다.
- 겹치는 한 쌍 중 편차(`max_deviation_mm`)가 더 큰 쪽을 대표로 삼아, 그 kind를
  `"복합(휨+끊김 의심)"`으로 명시적으로 재태깅한다. `DetectedDefect`에 `note` 필드를 추가해
  원래 두 후보의 편차 수치를 함께 남긴다(예: "같은 위치에서 휨(편차 8.6mm)과 끊김(편차 8.16mm)이
  동시 검출되어 편차가 더 큰 쪽(휨)을 대표값으로 병합함").
- 시각화(`visualize()`)에도 `"복합(휨+끊김 의심)"` 전용 색상(자홍색)을 추가해 콘솔·JSON·PNG
  세 곳 모두에서 이 태그가 드러나도록 했다.

**회귀 검증 (case0~6, 실제 재실행)**

이 후처리 때문에 case1(휨 단독), case3·case5(휨+끊김 복합 중 겹치는 부분), case6(휨 2개 동시)의
검출 결과 kind가 `"복합(휨+끊김 의심)"`으로 바뀌었다. 기존 `main()`의 자체 검증 로직(주입 결함
유형이 검출 결과에 포함되는지 확인)은 문자열이 정확히 같아야 통과하도록 되어 있었는데, 이 로직도
"복합" 태그가 원래 휨/끊김 어느 쪽이 주입되었어도 매칭된 것으로 인정하도록 함께 갱신했다(`
_kind_matches()` 헬퍼 추가). 또한 `detection_count_by_kind` 집계도 복합 태그를 휨/끊김 양쪽
카운트에 모두 반영하도록 수정했다(그렇지 않으면 case6처럼 다중 휨이 전부 복합으로 재태깅될 때
다중결함 개수 검증이 거짓으로 실패했다 — 최초 수정 시 실제로 이 문제가 발생해 재수정함).

실제 재실행 결과, case0~6 전체 **PASS**:

```
자체 검증 요약 (주입 결함 유형이 실제로 검출되었는가)
======================================================================
  case0_normal: 주입=없음 검출=없음 -> PASS
  case1_bend_only: 주입=['휨'] 검출=['복합(휨+끊김 의심)'] -> PASS
  case2_break_only: 주입=['끊김'] 검출=['끊김'] -> PASS
  case3_bend_and_break: 주입=['끊김', '휨'] 검출=['끊김', '복합(휨+끊김 의심)'] -> PASS
  case4_multi_break: 주입=['끊김'] 검출=['끊김'] -> PASS (다중결함 개수검증: PASS)
  case5_multi_break_and_bend: 주입=['끊김', '휨'] 검출=['끊김', '복합(휨+끊김 의심)'] -> PASS (다중결함 개수검증: PASS)
  case6_multi_bend: 주입=['휨'] 검출=['복합(휨+끊김 의심)'] -> PASS (다중결함 개수검증: PASS)

전체 결과: PASS
```

**이번 수정으로 해결하지 않은 것**

이 후처리는 "같은 위치의 휨/끊김 중복 표기"만 정리한다. `spec.md` 5절이 요구하는 정식 라벨링
스키마상 "복합" 유형을 실제로 어떻게 취급할지(별도 클래스로 둘지, 항상 우세한 쪽으로 강제
단일화할지 등)는 실제 검사자 워크플로우·라벨링 정책이 확정된 뒤 재검토가 필요하다(P1 이후,
G6~G10 검사자 UI/HITL 구현과 연계될 사안). bbox가 겹치는지 여부만으로 "같은 위치"를 판단하는
현재 방식도 단순화된 것으로, 더 정교한 거리 기반 판단(예: 곡선을 따른 호 위치 근접도)이 실데이터
확보 후 필요할 수 있다.
