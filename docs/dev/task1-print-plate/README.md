# 과제① 인쇄판 문안검사 — 파이프라인 프로토타입

실제 인쇄물/디자인 원본 이미지가 아직 없으므로, `spec.md` 1절의 공통 파이프라인
(`[전처리] → [정합(Registration)] → [차분/이상탐지] → [판정·시각화]`) 중
**"정합 + 픽셀 차분"** 부분이 원리적으로 동작함을 코드로 직접 검증하는 실행 가능한
프로토타입이다. 합성 라벨 이미지 2장(정답본/불량본)을 코드로 생성하고, 호모그래피 기반
정합 후 차분으로 불일치 위치를 검출한다.

> **가정**: 실제 인쇄물 이미지가 없으므로 화장품 라벨 형태의 합성 텍스트 이미지를
> `generate_labels.py`로 직접 생성해 사용한다. 결함 유형·용어는 `problem.md` 1절의
> "과제① 구체적 실패 사례"(오탈자 "토너"→"토노", "나이아신아마이드"에서 '아' 누락,
> 마침표 누락, 잉크 번짐)를 그대로 재현했으며, `spec.md` 2절의 최우선 치명 클래스인
> "성분표시오류"(성분 항목 누락)도 함께 주입했다. 임의의 새로운 결함 유형은 만들지 않았다.

## 무엇을 검증하는가

1. **정합(Registration)이 실제로 오검출을 줄이는가** — 실제 촬영/인쇄 공정에서는
   완벽히 겹치는 두 이미지를 얻을 수 없다 (`spec.md` 3.3: "정렬 오차로 인한 오검출
   제거의 핵심"). 이를 재현하기 위해 불량본에 미세한 회전(1.5°)·이동·스케일 오차를
   추가로 부여했다 (`spec.md` 3.2 준수: 과제①은 ±1~2도 이내 소회전만 허용).
   → 정합 없이 바로 차분하면 이미지 전체가 어긋나 보여 오검출/과검출이 폭증하고,
   ORB 특징점 기반 호모그래피로 정합한 후에는 실제 문안 결함 위치만 남는 것을 보인다.
2. **차분(Diff)이 문안 결함 위치를 실제로 특정하는가** — 정합된 두 이미지의 그레이스케일
   차분 → 강건한(robust) 임계값 이진화 → 모폴로지 정리 → 연결요소 bbox 추출로 결함
   후보 영역을 얻는다 (버그 수정 이력: 과거엔 Otsu 자동 임계값을 썼으나 불안정성
   문제로 대체했다. 아래 "버그 수정 이력" 절 참고).
3. **재현율(Recall) 관점의 평가** — `spec.md` 6절 "치명 클래스 미검출 0"이 최우선
   지표이므로, 주입한 결함 각각이 실제로 검출되었는지를 최우선으로 보고한다.

`output/20_pipeline_detections.png`를 열어보면, 정답본과 다른 두 라인(전성분 라인,

정합을 하지 않아 라벨 테두리 전체가 오검출 영역으로 잡힌다 — **정합 단계가 왜 필수인지를
그대로 보여주는 결과**다.

## 중요한 발견: 왜 "점누락(마침표)"은 놓쳤는가 — spec.md가 "Diff만으론 부족하다"고 명시한 이유의 실증

`점누락` 결함만 검출에 실패했다. 원인을 직접 추적한 결과:

- 마침표 하나가 사라지며 만드는 픽셀 diff는 실제로 존재한다 (Otsu 이진화 직후 해당
  영역에 117개의 흑백 diff 픽셀이 남아 있음을 확인함).
- 하지만 호모그래피 정합이 완벽하지 않아(서브픽셀 단위 잔차 오차) 문장 전체에
  1~2픽셀 크기의 미세한 안티앨리어싱 diff 잡음이 깔려 있고, 그 잡음의 크기가
  마침표 하나가 만드는 diff 신호와 거의 같은 크기다. 이 잡음을 제거하기 위한
  모폴로지 연산(Opening)이 실제 마침표 신호까지 함께 지워버린다.
- 모폴로지 Opening을 빼면 잡음이 그대로 살아나 문장 전체에 300개 이상의 잡티가
  생기고, 그 안에 마침표 신호가 묻혀 구분이 불가능해진다.

이는 **우연한 버그가 아니라, 순수 픽셀 차분만으로는 점·기호처럼 작은 결함을
안정적으로 검출하기 어렵다는 구조적 한계**이며, 이 프로젝트 문서들이 이미 예견한
바와 정확히 일치한다.

- `spec.md` 1절: 과제①의 핵심 기술을 "이미지 정합 **+ OCR** + 픽셀 차분"으로
  명시 — Diff 단독이 아니라 OCR(문자 인식)과의 결합을 전제로 설계되어 있다.
- `problem.md` 1절: "미세 결함 취약 — 점·기호 누락 ... 은 육안으로 놓치기 쉽다"고
  명시 — 사람도 놓치기 쉬운 결함이라는 점을 이미 문제로 정의해 두었다.

즉 이번 프로토타입은 "정합+Diff만으로는 미세 결함(점누락)에서 재현율 100%를
보장할 수 없다"는 것을 실측으로 확인했고, 이는 `spec.md`가 OCR을 필수 구성요소로
설계한 근거를 뒷받침한다. (본 환경에는 OCR 엔진 바이너리(Tesseract 등)가 설치되어
있지 않아 이번 프로토타입에는 OCR 단계를 포함하지 않았다 — 아래 "확인 필요" 참고.)

## 확인 필요 (후속 검증 필요 사항)

- **OCR 결합**: 이번 실행 환경에는 Tesseract 등 OCR 바이너리가 설치되어 있지 않아
  (`pytesseract` 파이썬 패키지는 있으나 엔진 바이너리 부재), `spec.md`가 명시한
  "정합+OCR+Diff" 중 OCR 단계는 이번 프로토타입 범위에서 제외했다. 위 발견에서 보듯
  점누락처럼 Diff만으로 놓치는 결함을 잡으려면 문자 단위 OCR 비교(줄 단위 텍스트
  정렬 후 Levenshtein 등 문자열 diff)를 다음 단계에서 반드시 결합해야 한다.
- **실제 인쇄물/스캔 이미지로의 일반화**: 이 프로토타입은 합성 텍스트 이미지 기준으로
  동작을 검증한 것이며, 실제 인쇄물의 잉크 텍스처·조명 편차·해상도에서도 동일한
  파라미터(모폴로지 커널 크기, min_area, Otsu 임계값)가 유효한지는 실측 데이터로
  재검증이 필요하다 (`prompt.md` 한계사항: "촬영 환경이 표준화되지 않으면 성능 보장
  불가", "합성 데이터만으로 실전 배치를 확정하지 말 것").
- **라인 단위 GT의 근사성**: 이번 recall 평가는 결함이 포함된 "라인 전체"를 GT bbox로
  사용한 근사 평가다. 실제 프로젝트에서는 `spec.md` 5절 라벨링 스키마(불량 종류 +
  BBox + 텍스트 diff)에 따라 문자/토큰 단위 GT를 구축해 정밀 평가해야 한다.

## 알려진 한계 (이 프로토타입 자체의 한계)

- 합성 이미지 기반이므로 실제 인쇄·스캔 노이즈(잉크 텍스처, 먼지, 색 편차)는
  반영하지 않았다.
- ORB 특징점은 텍스트 위주 이미지에서도 잘 동작했지만(1328 inlier), 텍스트 밀도가
  매우 낮은 라벨(로고/그림 위주)에서는 특징점이 부족해 `registration.py`의 ECC
  대체 경로가 필요할 수 있다. 이 프로토타입은 두 경로 모두 구현해 두었으나 ECC
  경로는 이번 합성 이미지 조건에서는 실제로 트리거되지 않았다.
- OCR 미포함으로 인해 점누락처럼 매우 작은 결함의 재현율이 100%에 도달하지 못했다
  (위 "중요한 발견" 참고). **이는 최종 시스템이 아니라 원리 검증 프로토타입이므로,
  실전 배치 전 OCR 결합 및 실측 데이터 재검증이 반드시 필요하다.**

## 버그 수정 이력 (QA 리포트 대응, 2026-07-07)

`docs/qa/task1-print-plate-qa-report.md`(이하 "QA 리포트")의 [발견 1~3]에 대응해
`diff_detect.py`, `generate_labels.py`, `run_demo.py`를 수정했다. QA 범위 밖으로
명시된 OCR 결합·실측 이미지 재현성·정합(Registration) 강건성 관련 항목(QA 3.3절)은
이번 수정 범위에 포함하지 않았다.

### [발견 1] Otsu 자동 임계값의 불안정성 → 라벨 전체(93%) 오검출

**근본 원인**: `compute_diff_mask`가 `cv2.threshold(..., THRESH_OTSU)`로 diff 이미지
전체 히스토그램을 이미지마다 자동으로 2분할했다. 결함이 여러 개 겹쳐 강한 diff 신호가
많을 때는 임계값이 높게(69~71) 잡혀 정합 잔차 노이즈를 걸러냈지만, 결함이 1건뿐이거나
없을 때는 강한 신호가 부족해 임계값이 아주 낮게(9~14) 잡혔고, 그 결과 호모그래피의
서브픽셀 잔차·안티앨리어싱 노이즈까지 "결함"으로 잡혀 `merge_close_boxes`가 이를
연쇄 병합해 라벨 전체(약 93% 면적)를 덮는 거대한 단일 오검출 박스를 만들었다.

**수정 방법 (QA 제안 (b) 채택)**: `diff_detect.robust_threshold()`를 신설해 Otsu를
대체했다. 그레이스케일 diff 이미지는 배경(완전 일치) 픽셀이 90% 이상을 차지해 값이
정확히 0이므로, 전체 히스토그램 기준 median/MAD는 항상 0으로 붕괴해 쓸모가 없다.
대신 **diff > 0인 픽셀만** 모아 `median + k*1.4826*MAD` (k=6.0, 정규분포 근사
스케일 1.4826 적용)를 구했다. 이 부분집합(diff>0)은 "정합 잔차 노이즈 + 실제
결함 신호"가 섞여 있지만, 결함 신호는 그 안에서도 소수(outlier)이므로 MAD가 무너지지
않고 결함 개수와 거의 무관하게 안정적인 "노이즈 상한" 근처의 값을 낸다. 여기에 값이
붕괴/폭주하는 극단적 경우를 막기 위한 안전 하한(`fixed_min=30`)·상한(`fixed_max=200`)을
추가했다.

Otsu를 버리고 "고정 임계값(하한)"만 쓰는 대안(QA 제안 (a)/(b) 중 순수 고정값)도
검토했으나, 실측 이미지의 정합 잔차 노이즈 수준은 촬영 조건에 따라 달라질 수 있으므로
완전 고정값보다는 이미지별로 재조정되는 median+k*MAD 쪽이 더 일반적이라 판단해 (b)를
선택했다. 다만 median+k*MAD가 여전히 결함 신호에 일부 영향받을 수 있으므로, 이를
완전히 신뢰하지 않고 **오검출 면적 가드레일**(`oversize_area_ratio=0.20`)을 별도로
추가했다: 병합된 박스가 참조 이미지 면적의 20%를 넘으면 `info["oversized_flags"]`에
`True`를 표시한다. 박스를 억제하거나 숨기지 않고 "이 검출은 위치를 좁히지 못해
신뢰할 수 없다"는 사실을 지표에 그대로 드러내는 방식을 택했다 (완전 억제는 진짜
결함이 라벨 대부분을 뒤덮는 극단적 케이스를 숨길 위험이 있기 때문).

**실측 재현 (수정 전/후 비교)**:

| 케이스 | 수정 전 (Otsu) | 수정 후 (robust threshold + 가드레일) |
|---|---|---|
| 결함 없음(정합오차만 존재) | 임계값 9 → 박스 1개, 라벨 전체(93% 면적) 오검출 | 임계값 64.4 → **박스 0개** |
| 성분표시오류만 단독 발생 | 임계값 9~14(추정) → 박스 1개, 라벨 전체(93%) 오검출 | 임계값 64.4 → 박스 2개, 각각 면적비 0.1%/6.9% (오검출 가드레일 미발동, 정상 국소 검출) |
| 오탈자만 단독 발생 | (미측정, 동일 패턴 예상) | 임계값 54.5 → 박스 1개, 면적비 0.0% (결함 위치에 정확히 국소화) |
| 오탈자+문자누락+성분표시오류 동시(README 데모, 회귀 확인용) | 임계값 71 → 박스 2개, 검출 성공 | 임계값 74.3 → 박스 2개(면적비 0.8%/1.3%), 검출 성공 — **회귀 없음** |

가드레일 자체가 실제로 큰 오검출을 잡아내는지도 별도로 확인했다: `robust_threshold`를
의도적으로 무력화(`fixed_max=10`)해 옛 Otsu(9) 버그를 재현시키면, 박스 1개·면적비
93.3%·`oversized_flags=[True]`로 정확히 플래그가 올라오는 것을 확인했다 (가드레일이
독립적으로도 정상 동작함을 별도 검증).

### [발견 2] 라인 단위 GT 근사 → 결함 유형별 독립 검증 미보장

**근본 원인**: `00_ground_truth.json`의 `defect_line_boxes`가 라인(`ingredient`)
단위였고, 오탈자·문자누락·성분표시오류 3개 결함이 모두 같은 줄에 있어 `evaluate_recall`이
하나만 diff에 걸려도 그 라인에 속한 결함 전부를 "검출됨"으로 표시할 수 있었다.

**수정 방법**: `generate_labels._measure_substring_bboxes()`를 신설해, 결함이
주입되기 전(=정답본과 동일한) ingredient 원본 텍스트에서 각 결함이 차지하는
부분 문자열("토너", "나이아신아마이드", ", 향료.")의 x범위를 PIL 폰트 메트릭
(`draw.textlength`)으로 직접 측정했다. 측정 기준을 정답본 텍스트로 고정한 이유는,
diff는 항상 참조(정답본) 이미지 좌표계에서 계산되므로 GT도 같은 좌표계를 기준으로
잡아야 좌표가 일치하기 때문이다. 결과 bbox는 `injected_defects[i]["bbox"]`에
결함 단위로 저장되며, `run_demo.evaluate_recall`은 이 결함 단위 bbox를 우선
사용하고 없을 때만(예: usage/caution처럼 원래 1라인=1결함인 경우) 라인 단위
GT로 폴백한다. 완전한 문자/토큰 단위 정밀 GT(예: 다른 프로젝트의 2인 교차검수
라벨링 스키마 수준)는 아니지만, "같은 줄에 겹친 결함들을 서로 구분해서 채점한다"는
발견 2의 핵심 문제는 해소했다.

**실측 재현**: 수정 전에는 ingredient 라인의 3개 결함(오탈자/문자누락/성분표시오류)이
모두 동일한 overlap_score=0.563(라인 전체 bbox 기준)으로 나왔다. 수정 후에는
결함별로 overlap_score가 0.958/0.704/0.828로 서로 다르게 나온다 — 즉 실제로
diff가 어느 결함 위치를 얼마나 정확히 덮었는지가 결함별로 구분되어 나타난다.

### [발견 3] Precision/오검출 면적 지표 부재

**수정 방법**: `run_demo.compute_precision_proxy()`를 추가했다. 문자 단위 실측
Precision(전체 diff 픽셀 중 실제 결함 픽셀의 비율)은 문자 단위 GT가 없어 정확히
계산할 수 없으므로, "검출된 박스 면적 중 실제 결함 GT 영역과 겹치는 비율"을
근사 지표로 사용했다:  `precision_proxy = (검출 박스 ∩ GT 영역 합) / (검출 박스 전체 면적)`.
라벨 전체를 덮는 오검출 박스처럼 GT와 무관한 영역을 많이 포함할수록 이 값이
낮아지므로 발견 1과 같은 실패를 recall과 독립적으로 드러낸다. 또한 [발견 1]에서
설명한 오검출 면적 가드레일(`oversize_area_ratio`, `any_oversized`)도 `run_demo.py`
STEP 4/5 출력과 `99_summary.json`에 그대로 노출한다.

**실측 (회귀 데모 케이스)**: `precision_proxy = 62.1%` (검출 총면적 22,907px² 중
GT와 겹친 면적 14,214px²). 100%가 아닌 이유는 각 결함 GT bbox가 문자 단위로 타이트하게
측정된 반면, 검출 박스는 모폴로지 연산(Close/Open)과 `merge_close_boxes`로 약간의
여유 패딩을 두르기 때문이다 — 이는 정상적인 동작이며, 발견 1처럼 "GT와 무관한 라벨
전체를 덮는" 경우와는 값의 크기(수십 %대 vs 이론상 1~수% 미만)로 구분된다.

### 회귀 확인 요약

위 표와 같이 기존 README 데모 케이스(오탈자+문자누락+성분표시오류+점누락+번짐 동시
주입)를 재실행한 결과 **Recall 80%(4/5건), 치명 클래스(성분표시오류) 검출 성공은
수정 전/후 동일하게 유지**되며, 신뢰 가능 검출(`reliable_recall`)도 80%로 동일하다.
즉 이번 수정은 기존에 정상 동작하던 시나리오를 깨지 않으면서, QA가 지적한 "단독/무결함
케이스에서의 전면 오검출"만 해소했다.
