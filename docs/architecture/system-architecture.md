# 시스템 아키텍처 — 인쇄 품질검사 AI (인쇄판 문안검사 / 목형 칼날검사)

> 근거 문서: `intro.md`, `problem.md`, `spec.md`, `prompt.md`. 본 문서는 위 문서들과 상충되는 내용을 임의로 정하지 않으며, 미해결 사항(spec.md 8절)에 대해서는 "가정:"으로 명시한 뒤 설계한다.

## 0. 설계 목표 정렬 (prompt.md 근거)

- 미검출(재현율) 최우선, 오검출은 재검 프로세스로 흡수 (prompt.md 목표 1)
- 불량 데이터 희소성을 합성 증강(①) / 이상탐지(②)로 우회 (목표 2)
- 정합(Registration) 정확도가 전체 파이프라인의 전제조건 (목표 3)
- 판정 결과는 좌표/mm 단위 위치오차 + 근거 시각화 포함, MES 연계로 추적 가능 (목표 4, spec.md 7절)
- 성분표시(법적 리스크) 등 치명 클래스는 AI가 단독 확정하지 않고 사람 재검 전제 (prompt.md 제약사항) → 아키텍처상 "AI 판정"과 "최종 출하 판정"을 분리된 컴포넌트/권한으로 설계해야 함
- 목표 성숙도: "기초 → 중간1" — 완전 무인 자동판정 시스템이 아님 (Non-goals) → HITL(Human-in-the-loop) 재검 워크플로우를 1급 컴포넌트로 포함

## 1. 공통 파이프라인 (spec.md 1절 그대로 채택)

```
[촬영/입력] → [전처리·정규화] → [정합(Registration)] → [차분/이상탐지] → [판정·시각화] → [결과 기록(MES 연계)]
```

두 과제(①인쇄판 문안검사, ②목형 칼날검사)는 이 6단계 파이프라인을 공유하되, 각 단계의 구현(알고리즘·모델)은 과제별로 분기한다. 아키텍처는 **단계(Stage) 인터페이스는 공통, 구현(Implementation)은 과제별 플러그인**으로 설계한다.

### 1.1 단계별 상세

| 단계 | 공통 인터페이스 | 과제① 구현 | 과제② 구현 |
|---|---|---|---|
| ① 촬영/입력 | `RawCapture{image, metadata}` | 스캐너/라인스캔 카메라, 600dpi+, ICC 색프로파일 | 고정 지그, 백라이트/텔레센트릭 렌즈, 편광 |
| ② 전처리·정규화 | `NormalizedImage` | 데스큐, 배경 제거, 해상도/명암/색 정규화 | 에지 추출(Canny), 반사 억제(편광), 스케일 캘리브레이션 |
| ③ 정합(Registration) | `AlignedPair{ref, target, transform}` | 특징점 매칭 + 호모그래피 | CAD 오버레이 + 스케일 캘리브레이션 |
| ④ 차분/이상탐지 | `DefectCandidates[]` | Diff + OCR(문자 인식/혼동쌍 보정) | 이상탐지(PatchCore/PaDiM) + 세그멘테이션 |
| ⑤ 판정·시각화 | `Verdict{class, bbox/mask, score, overlay}` | 불량종류+BBox+텍스트diff | 불량종류+세그멘테이션+위치오차(mm) |
| ⑥ 결과 기록(MES 연계) | `InspectionRecord` | 제품코드·LOT·판버전·좌표·불량유형 | 목형ID·제품·누적사용횟수·불량위치 |

이 표는 spec.md 1절·5절의 입출력 정의를 그대로 계승한 것이며, 신규 가정을 추가하지 않았다.

### 1.2 왜 "공통 인터페이스 + 과제별 플러그인" 구조인가

- intro.md의 "공통 원리"(정합 → 차분 → 판정)를 아키텍처 레벨에서 코드 재사용 가능하게 만든다.
- 향후 3번째 검사 과제(예: 다른 인쇄물 유형)가 추가되어도 파이프라인 골격을 재사용할 수 있다.
- 각 단계는 독립적으로 교체·버전관리 가능해야 한다 (spec.md 7절 "버전 관리" 요구사항 — 원본/CAD 버전과 실 생산 버전 불일치 검증이 정합 단계의 전제조건이므로, 정합 단계 앞에 "버전 검증 게이트"를 명시적 서브스텝으로 둔다).

## 2. 컴포넌트 경계 (Component Boundaries)

```
┌───────────────────────────────────────────────────────────────────┐
│                         MES / ERP (외부 시스템)                     │
│              — 신규 구축 범위 밖. 기존 스키마·API 연계만 수행 —          │
└───────────────────────────────△──────────────────△─────────────────┘

│                      Inspection Orchestrator (검사 오케스트레이터)      │
```

### 2.1 컴포넌트별 책임

1. **Capture Gateway**: 촬영 지그/카메라 제어, 원본 이미지 메타데이터(제품코드, LOT/목형ID) 태깅. 촬영 표준화(spec.md 7절) 위반 시(조도/해상도 불일치) 입력 단계에서 거부.
2. **Preprocess Service**: 정규화, 데스큐, 에지 추출 등 과제별 전처리. 품질 게이트(초점불량·과露光 이미지 재촬영 요청) 포함.
3. **Registration Service**: 원본/CAD 대비 정밀 정렬 + **버전 검증 게이트**(원본/CAD 버전 vs 생산 버전 일치 확인 — 불일치 시 "데이터 관리 이슈"로 분류하고 모델 오검출과 구분해 알림).
4. **Defect Engine**: 과제별 모델 서빙. ①은 OCR+Diff 파이프라인, ②는 이상탐지+세그멘테이션 모델. 모델 버전/가중치는 독립 배포·롤백 가능해야 함.
5. **Verdict & Overlay Renderer**: 비개발자 검사자가 이해 가능한 시각적 비교("정답과 무엇이 다른지", prompt.md 목표 5) 생성. 성분표시오류·칼날끊김 등 치명 클래스는 시각적으로 강조(색상·우선순위 플래그).
6. **HITL Review Console**: 검사자가 AI 판정을 확인·재검·최종 확정하는 화면. 치명 클래스는 AI 단독 확정 금지 → 이 컴포넌트에서 사람 확정이 반드시 개입하는 워크플로우 강제(승인/반려/보류 상태 머신).
7. **Traceability Store**: 판정 결과(불량유형, 좌표/mm, 근거 이미지, 검사자 확정 이력)를 제품코드·LOT/목형ID와 함께 영구 저장. MES 조회 가능하도록 스키마 노출.
8. **Inspection Orchestrator**: 전체 흐름 제어, 단계 실패/재시도, 치명 클래스 검출 시 자동 출하 보류 라우팅.

### 2.2 경계 설계 원칙

- **MES는 소비자(consumer) 겸 데이터 제공자**로만 다룬다 (신규 MES 구축은 non-goal). Orchestrator ↔ MES 사이는 명확한 API 계약(스키마)으로 경계를 긋는다.
- **Defect Engine은 과제①/②를 별도 서비스로 분리**한다 (모델 특성이 근본적으로 다름: OCR 기반 vs 이상탐지 기반). 공통 파이프라인 골격은 공유하지만 배포 단위는 분리해 독립적 스케일링/재학습이 가능하게 한다.
- **HITL Review Console은 판정 엔진과 물리적으로도 분리**한다. AI가 "검출"까지만 하고 "최종 출하 확정"은 별도 컴포넌트(사람 권한)에서 이뤄지도록 하여, 법적 책임 소재와 시스템 경계를 일치시킨다.

## 3. 데이터 흐름 (Data Flow)

```
[원본/도면 저장소]──────┐
 (디자인원본 AI/PDF,      │  기준 버전 조회
  CAD 도면)               ▼
[MES: LOT/목형ID/버전 마스터] ──▶ Registration Service (버전 검증)
                                      ▲
[Capture Gateway] → RawCapture(이미지+메타) → Preprocess Service → NormalizedImage
                                                                        │
                                                                        ▼
                                                          Registration Service
                                                          (특징점/호모그래피 ①, CAD오버레이 ②)
                                                                        │
                                                                        ▼ AlignedPair
                                                              Defect Engine (①/②)
                                                                        │
                                                                        ▼ DefectCandidates[]
                                                        Verdict & Overlay Renderer
                                                          (치명클래스 플래그 + 시각화)
                                                                        │
                                          ┌─────────────────────────────┴───────┐
                                          ▼                                     ▼
                                 정상 판정 (Precision 낮아도 흡수)      치명/의심 판정 → 강제 재검
                                          │                                     │
                                          └───────────────┬─────────────────────┘
                                                           ▼
                                                  HITL Review Console
                                                  (검사자 확정: 승인/반려)
                                                           ▼
                                                  Traceability Store
                                                           ▼
                                                     MES 결과 기록
                                                (제품코드·LOT/목형ID·불량유형·좌표/mm·근거이미지)
```

핵심 데이터 흐름 규칙 (spec.md/prompt.md 근거):

- 모든 검사 결과는 제품코드·LOT/목형ID와 연계되어야 MES에서 조회 가능 (spec.md 7절 추적성).
- 판정 근거(불일치 위치 시각화)는 결과와 함께 반드시 저장·전달 (spec.md 7절 설명가능성).
- 치명 클래스(성분표시오류, 칼날끊김·휨) 검출 시 흐름이 "정상 판정" 경로를 타지 않고 강제로 HITL 재검 경로로 라우팅되어야 한다 (미검출 0 목표 + 사람 최종 확정 전제).
- 원본/CAD ↔ 실물 정합쌍은 버전 일치 확인이 정합(Registration) 이전에 선행되어야 한다 (버전 불일치 = 데이터 관리 문제, 모델 결함과 분리 집계).

## 4. 배포 방식: 엣지 vs 서버 추론 (spec.md 8절 미해결 사항 — 가정 명시)

spec.md 8절은 이를 "라인 속도·지연 요구사항에 따라 결정"할 미해결 사항으로 남겨두었다. 현재 두 문서 세트에는 라인 속도(초당 처리량)·네트워크 환경·설비 예산에 대한 구체 수치가 없으므로, 아래는 **가정**에 기반한 권고이며 실측 요구사항 확보 시 재검토가 필요하다.

**가정 1**: 두 과제 모두 "전수검사 자동화"가 목표이나(problem.md, 전수검사 어려움→샘플링 의존 문제 해결), 현재 목표 성숙도가 "기초→중간1"이며 완전 무인 자동판정이 아니므로(prompt.md non-goals), 실시간 컨베이어 속도로 밀리초 단위 판정을 강제하는 인라인(In-line) 100% 자동 차단 시스템까지는 요구되지 않는다고 가정한다.

**가정 2**: 촬영은 표준화된 지그(고정 조명·거리, spec.md 3.1)에서 이뤄지므로, 촬영과 판정이 동일 물리적 라인 속도에 실시간으로 묶여야 하는 제약은 상대적으로 낮다고 가정한다 (즉 오프라인/배치 검사도 허용 가능한 운영 형태로 가정).

이 두 가정 위에서 **권고: "서버(온프레미스 또는 사내 GPU 서버) 중심 추론 + 엣지는 캡처/전처리만" 하이브리드**를 1차 권고안으로 제시한다.

| 근거 | 서버 추론이 유리한 이유 |
|---|---|
| 모델 재학습 빈도 | spec.md는 합성 데이터 증강, 리얼-합성 갭 모니터링(8절)을 지속 요구 → 모델이 자주 갱신됨. 서버 중앙 배포가 버전 관리·롤백에 유리 |
| 모델 크기/연산량 | ②의 이상탐지(PatchCore/PaDiM)는 정상 데이터 임베딩 뱅크(메모리뱅크) 기반으로 연산·메모리 요구량이 있어 엣지 디바이스 제약 대비 서버가 유리 |
| 추적성/중앙 로그 | spec.md 7절 추적성(MES 연계, 판정근거 보관)은 중앙 저장소가 전제 → 서버 추론 시 원본 이미지·중간산출물을 별도 전송 없이 그대로 활용 가능 |
| HITL 재검 콘솔 연동 | 재검 콘솔은 여러 검사 지점의 결과를 통합 조회해야 하므로 중앙 서버 구조와 자연히 부합 |
| 초기 성숙도 단계 | "기초→중간1" 단계에서는 처리 속도보다 정확도·설명가능성 검증이 우선 (prompt.md 목표 1, 3) → 엣지 최적화(양자화 등에 따른 정확도 손실 리스크)를 초기에 감수할 필요가 낮음 |

**엣지가 필요해지는 조건(향후 재검토 트리거)**: 다품종 다지점 라인에 카메라가 다수 분산되어 네트워크 대역폭이 병목이 되는 경우, 또는 인라인 실시간 차단(라인 속도 동기 판정)이 사업 요구로 확정되는 경우. 이 경우 "전처리·정합(경량)은 엣지, 이상탐지/OCR 추론은 서버"로 재분리하는 하이브리드 재설계를 권고.

**확인 필요**: 실제 라인 속도(분당/시간당 검사 물량), 공장 네트워크 인프라(대역폭/지연), 검사 지점 수와 지리적 분산 여부. 이 수치가 확보되면 본 권고안을 재검증해야 한다.

## 5. 기술스택 후보

## 6. 미해결 사항 승계 (spec.md 8절)

아키텍처 설계 시점에는 아래 사항이 여전히 미해결이며, 각 팀 작업에 영향을 준다:

1. 실불량(NG) 샘플의 목표 최소 수량/확보 일정 — 미정. Defect Engine의 test셋 신뢰도, 모델 배포 승인 기준(release gate)에 영향.
2. 엣지 vs 서버 추론 — 본 문서 4절에서 "서버 중심"을 1차 가정 기반 권고로 제시했으나 최종 확정 아님. 인프라팀 확인 필요.
3. 리얼-합성 갭 모니터링 절차 — 미정. Traceability Store에 "합성 대비 실측 성능" 모니터링 대시보드 요구사항으로 이어질 수 있음(향후 설계 필요).
