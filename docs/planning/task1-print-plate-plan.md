# 기획 문서 — 과제① 인쇄판 문안검사 AI

> 근거 문서: `intro.md`, `problem.md`, `spec.md`, `prompt.md` (프로젝트 루트) 및 `docs/architecture/system-architecture.md`, `docs/data-analysis/data-strategy.md`. 본 문서는 위 문서들과 상충되는 내용을 임의로 정하지 않으며, 미해결 사항은 "가정:"으로 명시한다.

## 0. 과제 정의 요약 (근거: intro.md, problem.md)

- **목적**: OCR·비전으로 인쇄판(또는 1차 인쇄물) 이미지와 디자인 원본을 비교해 문안 정합성을 자동 검증 (intro.md §과제①)
- **검출 대상**: 문자누락, 오탈자, 점·기호누락, 번짐, 성분표시오류(누락/위치오류) (spec.md §2)
- **치명 클래스**: 성분표시오류 — 법적 리스크 직결, 미검출 0이 최우선 목표 (intro.md, prompt.md 목표1)
- **문제 유형**: 참조기반 이상검출 = Registration + OCR + Diff (spec.md §1, §4)

## 2. 검사자 사용 시나리오 (User Scenarios)

현재 목표 성숙도는 "기초 → 중간1"이며 완전 무인 자동판정이 아님(prompt.md Non-goals). 따라서 모든 시나리오는 AI 보조 + 사람 최종 확정 구조를 전제로 한다.

### 시나리오 A — 정상 LOT 검사 (다수 케이스)
1. 검사자가 인쇄판/1차 인쇄물을 지정 지그에서 스캔한다.
2. 시스템이 자동으로 디자인 원본과 버전을 확인(F-02)하고, 정합 후 Diff/OCR 검사를 수행한다.
3. 이상 없음(정상) 판정 시 결과가 오버레이 이미지와 함께 대시보드에 표시되고, 검사자는 결과를 확인 후 통과 처리한다.
4. 결과는 제품코드·LOT과 함께 MES에 자동 기록된다.

### 시나리오 B — 치명 클래스(성분표시오류) 검출 케이스
1. AI가 성분표시 영역에서 누락 또는 위치오류를 검출한다.
2. 시스템은 해당 LOT을 자동으로 "재검 대기열"로 강제 라우팅하고 출하를 보류한다 (F-07).
3. 검사자는 HITL 재검 콘솔에서 AI가 표시한 위치·원본 대비 차이를 시각적으로 확인한다 (F-08).
4. 검사자가 최종 판정(불량 확정/오검출 판단)을 내리고 사유를 기록한다 (F-14).
5. 오검출로 판단된 경우, 해당 케이스는 재학습용 피드백 데이터로 아카이빙된다.

### 시나리오 C — 비치명 결함(오탈자/점누락/번짐) 검출 케이스
1. AI가 비치명 클래스 결함을 검출하고 BBox+diff를 표시한다.
2. 해당 결과는 통상 재검 프로세스로 흡수되며(과검출 허용), 검사자가 최종 확인 후 통과/반려를 결정한다.
3. 반려 시 해당 불량은 좌표·유형과 함께 아카이빙되어 향후 실불량 데이터로 축적된다 (spec.md §3.1).

### 시나리오 D — 원본/버전 불일치 상황
1. 정합 단계 진입 전 버전 검증 게이트에서 원본 버전과 생산 버전 불일치가 감지된다.
2. 시스템은 이를 "AI 오검출"이 아닌 "데이터 관리 이슈"로 별도 표시하고, 검사자에게 원본 재확인을 요청한다 (prompt.md 제약사항).

### 시나리오 E — 라벨링/검수 담당자 워크플로우
1. 신규 정상/합성결함 데이터가 라벨링 대기열에 들어온다.
2. 2인 교차검수를 통해 불일치를 조정하고 라벨을 확정한다 (F-13).
3. 확정된 라벨은 train/val/test로 LOT 단위 분할되어 저장된다 (F-15).

## 3. 성공 기준 (Success Criteria — spec.md §6 반영)

| 구분 | 지표 | 목표 | 근거 |
|---|---|---|---|
| 최우선 지표 | 치명 클래스(성분표시오류) 재현율 | 미검출 0 지향 (Recall → 1.0에 근접) | spec.md §6, prompt.md 목표1 |
| 최우선 지표(전체) | 전체 클래스 재현율 | 미검출 최소화를 정밀도보다 우선 | spec.md §6 |
| 보조 지표 | Precision | 오검출은 재검 프로세스로 흡수 가능 → 상대적으로 낮은 우선순위지만 재검 부하 관리를 위해 모니터링 | spec.md §6 |
| 보조 지표 | F1, mAP | 클래스별 균형 확인용 | spec.md §6 |
| 운영 원칙 | 오검출 흡수 | 과검출은 재검 프로세스로 흡수, 미검출 절대 우선 최소화 | spec.md §6 |
| 성숙도 목표 | 사업계획 기준 | "기초 → 중간1" 수준 (완전 무인화 아님) | spec.md §6, prompt.md Non-goals |
| 검증 방식 | 리얼-합성 갭 검증 | 합성 데이터 학습 모델은 반드시 실불량 test셋으로 별도 검증, 합성만으로 배치 판단 금지 | prompt.md 제약사항, spec.md §8 |
| 데이터 품질 | 라벨 신뢰도 | 2인 교차검수 완료율 100% (P0 데이터셋 기준) | spec.md §5 |
| 추적성 | MES 연계 | 모든 검사 결과가 제품코드·LOT과 연계되어 조회 가능 | spec.md §7 |

**확인 필요**: 치명 클래스 재현율의 구체적 수치 목표(예: 99.x%)와 실불량 test셋 최소 샘플 수는 spec.md §8에서 미해결로 명시되어 있어, 파일럿 단계에서 실측 데이터 확보 후 정량 목표를 확정해야 한다. 본 기획서는 "미검출 0 지향"이라는 방향성만 채택하고 임의의 수치를 만들지 않는다.

## 4. 단계별 일정 초안 (Roadmap)

원칙: 불량 데이터 희소성이 근본 난제이므로(problem.md §2), 각 단계는 "데이터 확보 → 합성 증강 검증 → 실불량 소량 검증 → 사람 재검 병행 운영"의 순서를 따른다. 구체 기간(주/개월)은 spec.md §8의 미해결 사항(실불량 최소 확보량, 엣지/서버 결정)에 의존하므로, 본 초안은 **선후관계와 단계별 완료 기준(Exit Criteria)** 중심으로 작성한다.

**가정**: 구체적 캘린더 일수는 팀 리소스·실측 데이터 확보 속도에 따라 달라지므로, 아래는 "표준 소요 스프린트(1 스프린트=2주)" 단위의 상대적 추정치로 표기하며 확정 일정이 아니다.

### 단계 1 — 데이터 확보 (기반 구축)
- **작업**: 촬영 표준화(스캐너/라인스캔 지그, ICC 색프로파일 고정), 승인완료 인쇄판/1차출력물 정상 데이터 수집, 디자인 원본 라스터화 파이프라인 구축, 검사 반려품(실불량) 아카이빙 프로세스 수립, 라벨링 스키마·2인교차검수 체계 구축
- **Exit Criteria**: 제품·폰트·언어별 정상 데이터 다양성 확보, 촬영 지그 표준화 완료(재현성 검증), 실불량 아카이빙 프로세스 가동 시작
- **추정**: 3~4 스프린트
- **의존**: 없음 (선행 단계)

### 단계 2 — 프로토타입 (Registration + OCR/Diff 코어)
- **작업**: F-01(정합), F-02(버전검증 게이트), F-03(OCR Diff), F-05(혼동쌍 보정) 구현. 합성 결함 생성기(F-11) 초기 버전으로 학습 데이터 증강. 성분표시 전용 검출 로직(F-04) 우선 검증
- **Exit Criteria**: 소규모 정상+합성결함 데이터셋으로 파이프라인 End-to-End 동작 확인. 정합 오차가 후속 오검출의 주원인이 되지 않음을 검증(spec.md §7)
- **추정**: 4~6 스프린트
- **의존**: 단계1의 정상 데이터·원본 라스터화 완료

### 단계 3 — 파일럿 (소규모 실환경 검증 + HITL 병행)
- **작업**: F-06~F-09(판정 시각화, HITL 재검 라우팅, MES 메타데이터 연계) 구현. 소량 실불량 test셋으로 치명 클래스 재현율 실측. 리얼-합성 갭 모니터링(F-12) 최초 실행. 검사자 대상 UI 사용성 검증(시나리오 A~D)
- **Exit Criteria**: 치명 클래스(성분표시오류) 실불량 test셋 기준 재현율이 목표 방향(미검출 0)에 근접함을 확인. 검사자가 AI 판정 근거를 신뢰하고 재검 여부를 판단 가능함을 UAT로 확인(prompt.md 목표5). 라인 속도·지연 요구사항 실측하여 엣지/서버 여부 결정(spec.md §8 미해결사항 해소)
- **추정**: 4~6 스프린트
- **의존**: 단계2 코어 파이프라인 완료, 실불량 소량 확보(단계1에서 지속 누적)
- **확인 필요**: 파일럿 착수를 위한 실불량 최소 샘플 수 기준 — spec.md §8 미해결, 파일럿 계획 확정 전 데이터팀과 별도 합의 필요

### 단계 4 — 배포 (운영 전환)
- **작업**: F-10(번짐 검출), F-13~F-15(라벨링 워크플로우·피드백 루프 정식화), 운영 모니터링(리얼-합성 갭 정기 검증 F-12 상시화), MES 연계 정식 API 전환
- **Exit Criteria**: 실 라인 투입 후 일정 기간 병행운영(AI 판정 vs 사람 전수검사 비교)에서 미검출 사례 없음 확인. 재검 부하가 운영 가능한 수준임을 확인(정밀도 모니터링)
- **추정**: 지속 운영 + 4 스프린트 병행운영 기간
- **의존**: 단계3 파일럿 완료 및 Exit Criteria 충족

### 단계 이후 (P2 고도화, 배포 이후 지속)
- F-16(Copy-Paste 확장), F-17(다국어/폰트 확장), F-18(엣지 최적화, 단계3 결정 결과에 따라 착수)
