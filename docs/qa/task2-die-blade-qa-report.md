## 2. 실행 중 발견한 버그·한계 (구체적 근거 포함)

### 2-A. [치명] 동일 종류 결함이 2개 이상 있으면 일부가 렌더링 단계에서 소실되어 미검출된다

- **재현**: `breaks=[{start_frac:0.10}, {start_frac:0.40}, {start_frac:0.75}]` (끊김 3개 주입) 실행 결과 → **검출 1건만** (2건 미검출, 66% 미검출률).
- **근본 원인**: `inject_break()`의 폐곡선 랩어라운드 처리 로직
  ```python
  if len(runs) >= 2 and runs[0][0] == 0 and runs[-1][-1] == n - 1 and ...:
      runs[0] = runs[-1] + runs[0]; runs.pop()
  ```
  이 조건은 "원본이 닫힌 폐곡선일 때 첫 컷"에서만 유효하다(폐곡선을 한 번 자르면 열린 호 하나가 되는 것이 맞음). 그런데 `run_scenario()`가 **두 번째 이후 끊김을 이미 끊긴(열린) 조각에 다시 적용**할 때도 동일 조건이 우연히 참이 되어(열린 호의 양 끝이 로컬 인덱스 0/n-1이므로), 이미 분리되어 있어야 할 두 조각을 다시 하나로 이어붙인다. `render_polylines()`는 이어붙여진 배열을 순서대로 직선으로 연결해 그리므로, 실제로는 끊어져 있어야 할 자리가 **직선으로 다시 메워져(bridging) 시각적으로 사라진다.**
- **검증**: 3회 컷 후 `len(polylines)`가 계속 1로 유지됨(정상이라면 3, 즉 3개의 분리된 열린 호가 나와야 함)을 확인. 렌더링된 마스크의 연결요소 개수도 1개(원래는 최소 3개 이상이어야 함)로 확인, 이미지로도 육안 확인함(첨부 로그 참고).
- **부가 문제**: `start_frac`이 매번 "원본 도면의 전체 둘레 기준"이 아니라 "그 시점에 남아있는 조각(잘려서 재인덱싱된 배열)의 로컬 인덱스 기준"으로 해석되어, 두 번째·세 번째 결함의 실제 주입 위치가 호출자가 의도한 위치와 다르다(위치 자체도 부정확).
- **영향도**: spec.md 6절 "결함 재현율(끊김·휨 미검출 = 0)" 목표와 정면으로 충돌. 실제 목형은 한 곳이 아니라 여러 곳이 동시에 파손/절단될 수 있으므로, **이 버그는 현재 프로토타입이 실무에 그대로 옮겨질 경우 안전에 직결되는 결함**이다. 다만 데모 자체의 4개 내장 시나리오는 모두 끊김이 최대 1개씩만 주입되어 이 경로를 타지 않았기 때문에 README의 "PASS" 결론에는 영향이 없다 — 즉 **이 버그는 데모의 자체 검증으로는 발견되지 않는 사각지대**였다.
- **수정 방향(제안)**: 조각이 이미 열린 호인지(전체 곡선의 부분집합인지) 여부를 명시적으로 추적하고, 랩어라운드 병합은 "최초 컷(원본이 닫힌 곡선일 때)"에만 적용하도록 플래그를 두어야 한다. 또한 `start_frac`을 매번 원본 전체 인덱스 기준으로 환산하도록 수정해야 한다.

### 2-B. [경고] ECC 정합이 큰 회전 오차(약 40° 이상)에서 대칭성으로 인해 잘못된 지점에 잘못 수렴하고, `registration_converged=True`가 이를 감지하지 못한다

- **재현**: 동일 끊김 1개를 주입한 채 회전 오차를 1.4°→90°까지 늘려 테스트.

  | 실제 회전 | ECC 추정 회전 | reg_ok | 검출 개수 |
  |---|---|---|---|
  | 1.4°~35° | 정확히 일치(오차 <0.01°) | True | 1건 (정상) |
  | **45°** | **-0.85°** (완전히 틀림) | **True** | **17건** (거짓 양성 폭증) |
  | 90° | 0.48° (완전히 틀림) | True | 4건 |

- **근본 원인**: 라운드 사각형은 180° 근사 대칭을 가지므로, ECC(MOTION_EUCLIDEAN)가 실제 회전 대신 지역 최솟값(대칭에 의한 착각 정렬)에 수렴한다. `findTransformECC`는 수치적으로 "수렴은 했으나 틀린 답"인 경우도 예외 없이 성공을 반환하므로, `registration_converged=True`라는 필드만으로는 정합 품질을 신뢰할 수 없다.
- **영향도**: README도 "확인 필요"로 이미 지적한 항목("큰 회전/스케일 오차·조명 반사 노이즈에서는 별도 강건성 검증 필요")과 일치하는 실측 근거. 실제 촬영 지그가 설계대로 ±수도 이내로 고정된다면 발생 확률은 낮지만, 지그 안착 실수·이물질 등으로 큰 오정렬이 생기면 **오검출 폭증(운영 부담)** 을 유발한다. 미검출 방향으로 작용했다는 증거는 이번 테스트에서는 없었으나(오히려 과검출 방향), 정렬이 완전히 틀어진 상태에서 실제 결함이 무작위 위치의 노이즈에 묻혀 놓치는 시나리오는 배제할 수 없다.
- **수정 방향(제안)**: 정합 후 잔차(예: 정합된 두 마스크 간 IoU 또는 평균 최근접거리)를 별도로 계산해 임계치 이상이면 "정합 실패/재검 필요"로 판정을 보류하는 가드레일 추가 필요.

### 2-C. [경미/설계 트레이드오프, 재확인 필요] 최소 결함 길이 필터(2.0mm)가 헤어라인 결함을 원천적으로 배제한다

- `detect_defects()`는 연결요소의 bbox 장변이 `MIN_DEFECT_ARC_MM(2.0mm)` 미만이면 무조건 노이즈로 간주해 버린다. 이는 "과검출 억제"를 위한 의도된 설계이지만, spec.md·problem.md 어디에도 "2mm 미만은 치명적이지 않다"는 근거가 없다(README도 "가정"으로 명시). 실제 목형 공차 스펙 확보 전까지는 **이 값이 실제 치명 결함(예: 미세 크랙)을 놓칠 수 있는 파라미터라는 점을 재확인해야 한다.**
- 테스트: 실제 편차 0.8mm(허용오차 1.0mm 미만) 휨은 설계상 정상 범위로 미검출 처리됨 — 이는 허용오차 설계상 올바른 동작. 반면 편차는 크지만 길이가 짧은 헤어라인성 결함(예: 편차 3mm, 길이 1.5mm)이 실존한다면 현재 로직상 무조건 필터링된다 — 이런 경우가 실제로 "치명"으로 분류되어야 하는지는 확인 필요.

### 2-D. [경미] 재현성·환경 이슈 (버그는 아니나 QA 중 확인된 리스크)
- `requirements.txt`에 버전 고정이 없어(numpy/opencv-python/pillow), 향후 라이브러리 메이저 업데이트 시 ECC/dilate 등 동작이 달라질 위험이 있음(현재 환경: opencv 4.13.0, numpy 2.4.2, pillow 11.3.0에서는 정상 동작 확인).
- 한글 경로(`cv2.imwrite` 무음 실패) 및 한글 폰트(`malgun.ttf` 부재 시 폴백) 이슈는 README가 이미 정확히 인지·문서화하고 있으며, 실행 중 실제로 해당 우회 코드가 정상 동작함을 확인함(별도 신규 이슈 없음).

## 3. 향후 실데이터 확보 시 추가로 필요한 테스트 케이스

우선순위는 spec.md 6절(끊김·휨 미검출 0)과 §2 버그를 반영해 정렬함.

1. **다중 동시 결함 케이스 (최우선, §2-A 회귀 검증)**
   - 동일 목형에 끊김 2개 이상 / 휨 2개 이상 / 끊김+휨 각 2개 이상 동시 발생
   - 두 결함이 서로 인접(연결요소가 겹치거나 붙는 거리)한 경우 vs 멀리 떨어진 경우
2. **실제 카메라/조명 조건의 이진화 품질**
   - 백라이트 실루엣 촬영의 실제 그레이스케일 이미지 → 이진화 이후 라인 두께·끊김·잡티(먼지, 반사)가 있는 실측 이미지로 파이프라인 재실행
   - 조명 불균일로 인한 가짜 끊김(그림자로 인한 국소 저대비) 케이스
3. **정합 강건성 경계값 확정**
   - 실제 지그의 안착 오차 스펙(예: ±X°, ±Ymm)을 확인한 뒤, 그 범위의 상한·하한 근처에서 정합 성공률 측정
   - 대칭 형상(정사각형에 가까운 목형, 원형 목형 등)에서 ECC 대칭 오수렴 재현 여부 (§2-B 확장)
   - 정합 실패/오수렴을 자동 감지하는 잔차 기반 가드레일의 임계치 튜닝
4. **마모(연속량) 케이스 — 이번 데모 범위 밖, 별도 프로토타입 필요**
   - 등급(정상/주의/교체) 또는 회귀 라벨과 실측 마모 이미지 페어 확보 후 별도 검증
5. **위치오차 케이스 — 이번 데모 범위 밖**
   - 목형 전체가 기준 위치에서 평행이동/회전된 경우(부분 변형이 아닌 전체 배치 오차)와, 정렬(Registration) 단계에서 이를 "정합 대상 오차"로 흡수해버려 실제 위치오차 결함을 놓치지 않는지 검증(현재 파이프라인은 정합으로 정렬 오차를 제거한 뒤 diff를 보므로, 목형 자체의 위치오차 결함과 촬영 정렬오차를 구분하지 못할 근본적 설계 이슈가 있을 수 있음 — **확인 필요**)
6. **최소 결함 크기 기준 확정 (§2-C)**
   - 실제 다이커팅 공정에서 "재작업/불량으로 이어지는" 최소 끊김 길이·최소 휨 변위를 목형 공차 스펙에서 확보한 뒤 `MIN_DEFECT_ARC_MM`/`DEFECT_TOL_MM` 재조정 및 그 경계값 재검증
7. **mm/px 캘리브레이션 실측 검증**
   - 실제 촬영 지그의 스케일(현재 가정 0.2mm/px)을 알려진 기준자(calibration target)로 측정해 대체하고, 그 값 변경이 검출 결과(특히 경계값 근처 결함)에 미치는 영향 재확인
8. **클래스 혼동 규칙 명확화 (README도 인지한 이슈)**
   - 휨 결함이 부수적으로 "끊김"으로도 동시 검출되는 현상에 대해, 실제 라벨링 스키마(spec.md 5절)상 최종 결함 유형을 어떻게 단일화할지 규칙 확정 후 재검증
9. **대량/반복 실행 시 성능·안정성**
   - 실제 라인 검사 처리량(초당 목형 수) 요구사항에 맞춘 처리시간 측정 — 이번 데모는 소수 시나리오만 실행해 처리시간 미측정
10. **test셋 누수 방지 확인**
    - 실불량 확보 후 spec.md 3.3 "목형ID·제품 단위 분할" 원칙이 실제로 지켜지는지, 합성 데이터로 사전 튜닝된 임계값(1.0mm, 2.0mm 등)이 실불량 test셋에서 과대평가되지 않는지 별도 검증

## 4. 결론 요약

- 프로토타입은 spec.md가 요구하는 파이프라인 **원리**(정합→diff→판정, mm 출력, 시각화)를 단일 결함 케이스에 대해서는 정확히 재현했다. README의 "PASS" 주장은 재실행으로 확인됨.
- 그러나 QA 중 **spec.md 6절 핵심 목표(끊김 미검출 0)에 직접 반하는 버그(§2-A, 동시 다중 결함 시 최대 66% 미검출)** 를 발견했다. 이는 데모의 4개 내장 시나리오가 모두 결함 1개씩만 주입해 우연히 회피된 사각지대이며, 실제 목형(다중 파손 가능)에는 그대로 적용할 수 없는 상태다.
- 추가로 대각도 오정렬 시 정합이 대칭성으로 오수렴하면서 검증 없이 "성공"으로 보고되는 신뢰성 이슈(§2-B)를 확인했다.
- 결론: 이 프로토타입은 **"원리 검증(PoC)" 단계로는 유효**하나, §2-A 버그 수정과 §2-B 가드레일 추가 없이는 실데이터 기반 다음 단계(모델링/실배치)로 진행하기에 부족하다. 실데이터 확보 전 반드시 §2-A를 먼저 수정하고 다중 결함 회귀 테스트를 통과시킬 것을 권고한다.
