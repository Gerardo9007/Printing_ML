# QA 검증 리포트 — 과제① 인쇄판 문안검사 프로토타입

- 대상 코드: `docs/dev/task1-print-plate/` (`generate_labels.py`, `registration.py`, `diff_detect.py`, `imgio.py`, `run_demo.py`)
- 검증 방법: README에 기록된 실행 결과를 Bash로 직접 재실행해 재현 확인 + 코드를 직접 조작해 README에 없는 경계/실패 케이스를 추가 실행
- 근거 문서: `intro.md`, `problem.md`, `spec.md`(2절 검출 클래스, 6절 목표 지표), `prompt.md`(목표1: 치명 클래스 미검출 0 최우선)

## 1. spec.md 목표 지표 관점 — 검증되는 것 vs 검증되지 않는 것

### 검증되는 것 (실측 확인)

| 항목 | 확인 내용 |
|---|---|
| 정합(Registration)의 필요성 | 정합 없이 차분(대조군, STEP3)하면 회전/이동 오차 때문에 라벨 전체가 1개의 거대 오검출 영역으로 뭉개짐 → ORB+호모그래피 정합 후에는 후보 영역이 2개로 줄어듦. 정합 단계가 오검출을 줄인다는 **원리 자체**는 실측으로 확인됨 |
| ORB 특징점 정합의 기본 동작 | 텍스트가 충분한 이미지에서 ORB+RANSAC 호모그래피가 정상 동작(inlier 1300~1600대), 크래시 없음 |
| ECC 대체 경로의 실동작 | README는 "이번 합성 이미지에서는 ECC 경로가 트리거되지 않았다"고만 기록했을 뿐 실제 동작 여부는 미확인 상태였음. 본 QA에서 저텍스처(단일 사각형) 이미지로 직접 강제 트리거한 결과 `ecc_euclidean`이 정상 동작함을 확인. 완전 백지 이미지(특징점 0개)에서도 예외 없이 `none(정합 실패)`로 안전하게 폴백함을 확인 (크래시 없음) |
| 점누락(마침표) 미검출의 원인 | README가 주장한 "모폴로지 Opening이 마침표 신호까지 지운다"는 설명을 코드 추적으로 재확인 가능 (diff_detect.py의 MORPH_OPEN 단계) |
| 성분표시오류 등 텍스트 결함이 diff로 "포착 가능한 크기의 신호"를 만든다는 것 | 오탈자/문자누락/성분표시오류가 있는 라인에서 실제 픽셀 diff가 검출 임계값을 넘는 것을 확인 |

### 검증되지 않는 것 / 이번 프로토타입 범위 밖 (spec.md 6절 지표 기준)

| spec.md 요구 항목 | 이번 프로토타입 상태 |
|---|---|
| **치명 클래스(성분표시오류) 미검출 0** (최우선 지표) | **검증되지 않음.** 아래 2절 [발견 1]에서 보듯, 이번 데모의 "성분표시오류 검출 성공"은 파이프라인이 결함 위치를 안정적으로 좁혀서 찾은 결과가 아니라, 같은 줄에 함께 있던 다른 결함(오탈자·문자누락)이 우연히 Otsu 임계값을 밀어 올려준 부작용이다. 성분표시오류가 **단독으로** 발생하면(현실적으로 더 흔한 케이스) 동일 코드가 라벨 전체(면적 93%)를 하나의 거대 오검출 박스로 반환하며, 이는 사실상 "전수 재검사"와 동일해 실사용 가치가 없다 |
| OCR 결합 (spec.md 1절: "정합+OCR+Diff") | 미구현. Diff 단독으로는 점누락 같은 미세 결함을 놓친다는 것만 확인됨(README도 인정) |
| Precision / 오검출률 | 측정 지표 자체가 없음. `run_demo.py`는 recall만 계산하고 박스 크기·오검출 면적에 대한 페널티가 전혀 없어, "라벨 전체를 덮는 거대 박스"도 recall 계산상 결함일 뿐 문제로 드러나지 않음 |
| 실측 이미지에서의 재현성 | 합성 텍스트 이미지 1쌍(변형 파라미터 1개)만으로 검증되어, 실제 인�쇄 잉크 텍스처/조명 편차/해상도에서의 동작은 전혀 확인되지 않음 (README도 명시) |
| 라인 단위가 아닌 문자/토큰 단위 정밀 평가 | GT가 라인 bbox 단위 근사이므로, "문자 단위 정확한 위치를 찾았는가"는 검증되지 않음 |
| 클래스 불균형/합성-실측 갭 | 대상 아님 (spec.md 8절 미해결 사항, 실데이터 확보 후 과제) |

## 2. 실행 중 발견한 버그/한계 (README에 기록되지 않은 신규 발견)

### [발견 1 — 가장 중요] Otsu 자동 임계값의 불안정성으로 "치명 클래스 단독 발생" 시 전면 오검출 발생

**재현 방법**: `generate_labels.make_defective`에서 오탈자·문자누락 결함을 제거하고 성분표시오류(향료 누락)만 단독으로 주입 → 동일 정합(ORB inlier 1621, 정상 동작) 후 diff 실행.

**결과**: 검출된 bbox가 `(18, 18, 864, 1164)` 단 1개 — 라벨 테두리를 제외한 이미지 전체(약 93% 면적)를 덮는 박스. 실측 비교:

| 케이스 | Otsu 임계값 | diff 픽셀 수 | 최종 검출 박스 |
|---|---|---|---|
| README 데모(오탈자+문자누락+성분표시오류 동시 발생, 같은 줄) | 71 | 10,102 | 2개, 결함 줄에 정확히 대응 |
| 결함 전혀 없음(정합오차만 존재) | **9** | 50,610 | 1개, 라벨 전체(93% 면적) |
| 성분표시오류만 단독 발생 | (동일하게 낮음) | — | 1개, 라벨 전체(93% 면적) |

`compute_diff_mask`는 `cv2.threshold(..., THRESH_OTSU)`로 임계값을 이미지마다 자동 산출한다. 결함 텍스트가 만드는 강한 diff 신호가 여러 개 겹쳐 있을 때는 Otsu가 임계값을 높게(71) 잡아 정합 잔차 노이즈를 걸러내지만, 결함이 없거나 결함이 1건뿐일 때는 임계값이 낮게(9) 잡혀 호모그래피의 서브픽셀 잔차·안티앨리어싱 노이즈가 라벨 전체 텍스트 라인에서 diff로 잡히고, `merge_close_boxes`(gap=25)가 이를 연쇄적으로 병합해 거대한 단일 박스로 만든다.

**왜 심각한가**: 이 거대 박스는 성분표시오류가 있는 줄과 100% 겹치므로(overlap_fraction=1.0), `run_demo.evaluate_recall`의 임계값(0.05) 기준으로는 "검출 성공"으로 기록된다. 즉 **현재의 recall 지표는 이 실패를 전혀 잡아내지 못하며, README의 "성분표시오류 100% 검출" 결론은 이 함정을 인지하지 못한 채 내려진 것**이다. spec.md 6절이 요구하는 "치명 클래스 미검출 0"은 위치를 좁혀 사람이 재검할 수 있는 형태로 찾아야 의미가 있는데, 라벨 전체를 덮는 박스는 실질적으로 "모든 것이 의심됨 = 전수 재검사"와 같아 자동화의 가치가 없다. 대조군(STEP3, 정합 없음)이 만드는 오검출과 본질적으로 동일한 실패가, 정합을 정상적으로 마친 뒤에도 결함 조합에 따라 재발할 수 있다는 뜻이다.

### [발견 2] 라인 단위 GT 근사 평가가 결함 유형별 독립 검증을 보장하지 못함

`00_ground_truth.json`의 `defect_line_boxes`는 라인(예: `ingredient`) 단위이며, 오탈자·문자누락·성분표시오류 3개 결함이 모두 같은 `ingredient` 라인 안에 있다. `evaluate_recall`은 검출 박스가 해당 "라인 bbox"와 5% 이상 겹치면 그 라인에 속한 **모든** 결함을 개별적으로 "검출 성공"으로 표시한다. 즉 실제로는 셋 중 하나(예: 오탈자)만 diff에 걸렸어도 나머지 둘(문자누락, 성분표시오류)도 자동으로 "검출됨"으로 집계될 수 있어, 결함 유형별 독립적인 재현율 주장이 아니다. (발견 1과 결합하면 더 심각: 라인 전체를 덮는 오검출 박스 하나가 그 라인의 모든 결함을 "검출 성공"으로 만들어버릴 수 있다.)

### [발견 3] Precision/오검출 면적 지표 부재

`run_demo.py`는 recall(및 치명 클래스 미검출 여부)만 계산하며, 검출 박스의 크기·개수·오검출 면적에 대한 지표가 없다. spec.md 6절은 "오검출(과검출)은 재검 프로세스로 흡수 가능"이라고 명시했지만, 이는 오검출이 "일부 국소 영역"일 때를 전제로 한 운영 원칙이다. 발견 1처럼 오검출이 "라벨 전체"가 되면 재검 프로세스로 흡수 불가능한 수준이 되므로, 최소한 "검출 박스가 이미지 면적의 X% 이상을 덮으면 실패로 간주" 같은 가드레일이 지표에 없다는 것 자체가 한계다.

### [발견 4 — 경미] 코드/문서 사소 불일치

- `diff_detect.draw_boxes`의 `labels` 파라미터가 `run_demo.py`에서 전혀 사용되지 않음(항상 `#1,#2,...`로만 표시) — 결함 유형 라벨을 시각화에 넣을 수 있는 인터페이스가 있으나 활용되지 않고 있음. 기능 버그는 아니지만 "설명 가능성"(prompt.md 목표4/5) 요구사항 대응 관점에서 개선 여지.
- `generate_labels.py`가 `C:/Windows/Fonts/malgun.ttf` 절대경로에 하드코딩되어 있어 Windows/맑은고딕 폰트가 없는 환경(Linux CI 등)에서는 그대로 실행 불가 (프로토타입 목적상 치명적 결함은 아니나 이식성 한계로 기록).
- ORB `nfeatures=4000`, `min_matches=15`, morph 커널 5x5, `min_area=40`, `merge gap=25` 등 다수 파라미터가 이번 합성 이미지 1종에 맞춰 하드코딩되어 있으며, README도 "실측 데이터 재검증 필요"를 인정함 — 위 발견 1이 이 파라미터 민감성의 구체적 증거가 됨(같은 파라미터로도 결함 조합에 따라 결과가 정반대로 바뀜).

## 3. 실데이터 확보 시 추가로 필요한 테스트 케이스

### 3.1 발견 1(Otsu 불안정성) 직접 검증용 — 최우선

1. 성분표시오류(누락/위치오류) **단독** 발생 샘플(다른 결함과 동시 발생하지 않는 케이스)에서 검출 박스 크기가 결함 영역에 국한되는지 확인 — 라벨 전체 대비 검출 박스 면적 비율을 새 지표로 도입해 상한(예: 20%) 초과 시 "검출 실패"로 재분류
2. 완전 정상(무결함) 샘플을 정합만 거쳐 diff했을 때 오검출 박스가 0개인지(또는 면적 상한 이내인지) 확인 — 현재 코드 기준으로는 실패함(본 리포트 재현)
3. 결함 개수·조합(1개/2개/3개 동시 발생)별로 Otsu 임계값과 최종 박스 수·면적이 어떻게 달라지는지 스윕 테스트 → 임계값을 이미지별 자동 산출(Otsu)이 아니라 고정값 또는 강건한 통계량(예: MAD 기반)으로 대체할지 결정할 데이터 확보

### 3.2 spec.md 2절 검출 클래스별 커버리지

4. 문자누락 — 어절 중간/끝/문장 시작 등 위치별, 한글 자모 단위 누락 포함
5. 오탈자 — 형태적으로 유사한 혼동쌍(spec.md 4절: O/0, l/1, ㅇ/0) 포함, 폰트 크기별
6. 점누락 — 마침표 외 쉼표·따옴표·괄호 등 다른 기호 누락, 다양한 폰트 크기(특히 소형 폰트)에서 diff 신호 대 노이즈 비율
7. 번짐 — 실제 인쇄 잉크 번짐(스캔/촬영 실측 이미지) 강도별(경미/심각) 샘플
8. 성분표시오류 — 항목 누락뿐 아니라 순서 변경, 위치 오류(같은 텍스트가 다른 라인/좌표로 이동), 함량 표기 오류

### 3.3 정합(Registration) 강건성

9. 회전 각도 스윕(spec.md 허용 범위 ±1~2도 경계값 포함, 초과 시 정합 실패를 올바르게 감지하는지)
10. 텍스트 밀도가 매우 낮은 라벨(로고/그림 위주)에서 ORB 실패 → ECC 대체 경로가 실제로 트리거되고 정상 동작하는지 (본 QA에서 합성 저텍스처 이미지로만 확인, 실제 라벨 이미지 기준 재확인 필요)
11. 조명/노출 차이가 있는 두 이미지(원본 vs 실측 스캔) 간 정합 정확도 — 밝기 차이가 ORB 특징점 검출률에 미치는 영향
12. 완전히 다른 버전(예: 개정 전/후 원본)을 잘못 매칭했을 때 — "버전 불일치"(prompt.md 한계사항)를 정합 신뢰도(inlier 수)로 감지·경고할 수 있는지

### 3.4 실측 이미지 기반 재보정

13. 실제 스캔/촬영 인쇄물(600dpi 이상)로 Otsu 임계값, 모폴로지 커널 크기, min_area, merge gap 파라미터 전체 재튜닝 및 민감도 분석 — 합성 이미지 파라미터를 그대로 가져다 쓰지 않기
14. 잉크 텍스처·먼지·색 편차가 있는 정상 샘플 다수에서 오검출률(발견 3 지표) 측정 — "정상인데 얼마나 자주 어디를 오검출하는가"의 베이스라인 확보
15. OCR 결합 버전(Tesseract 등 설치 후) — 점누락류 미세 결함에 대해 Diff-only 대비 recall 개선 여부 A/B 비교

### 3.5 평가 지표/방법론 개선

## 요약

이번 프로토타입은 **정합이 대조군보다 오검출을 줄인다는 원리** 자체는 실측으로 재현·확인했으나, README가 결론으로 제시한 "치명 클래스(성분표시오류) 100% 검출"은 재검증 결과 **신뢰할 수 없는 결론**임을 확인했다. 핵심 원인은 Otsu 자동 임계값이 결함 조합에 따라 9~71로 크게 흔들리는 불안정성이며, 성분표시오류가 다른 결함 없이 단독으로 발생하면(오히려 더 현실적인 시나리오) 동일 코드가 라벨 전체(면적 93%)를 오검출하고, 현재의 recall-only 평가 지표는 이 실패를 "검출 성공"으로 잘못 집계한다. spec.md 6절의 "치명 클래스 미검출 0"을 실질적으로 검증하려면 (a) Otsu 대신 강건한 임계값 산출 방식, (b) 오검출 면적을 페널티하는 지표, (c) 라인이 아닌 결함 단위의 독립 GT가 다음 단계에서 반드시 필요하다.
